# 嵌入式硬件产品入门指南

> 写给产品经理的完整体系教程 —— 从"这是个什么东西"到"怎么把它做出来"

---

## 前言：这份指南是给谁看的

如果你是个产品经理，第一次接触嵌入式硬件产品，你最需要的不是学会写代码或画电路图，而是建立一套**心智模型**：知道一个硬件产品由哪些部分拼起来的、每个部分的成本与风险在哪里、以及当工程师跟你说"这个做不了"时，你该怎么判断是真做不了还是条件没谈拢。

这份指南不讲代码、不讲寄存器、不讲工具链安装。它保留技术细节，但只保留那些**你做产品决策时真正用得上的**。

---

## 第一章：嵌入式产品到底是什么

### 1.1 从一个智能插座说起

假设我们要做一个智能插座——能用手机 App 远程开关、能显示当前功率、能设置定时。这个产品拆开来看，需要这些：

```
         ┌──────────────────────────────────┐
         │          智能插座产品              │
         │                                    │
         │  ┌──────┐  ┌──────────┐  ┌────┐  │
         │  │ MCU  │  │ WiFi模块  │  │电源 │  │
         │  │ (大脑) │  │ (通信)   │  │模块 │  │
         │  └──┬───┘  └──────────┘  └────┘  │
         │     │        ┌──────────┐         │
         │     ├────────│ 继电器   │         │
         │     │        │ (执行器) │         │
         │     │        └──────────┘         │
         │     │        ┌──────────┐         │
         │     └────────│ 电流检测 │         │
         │              │ (传感器) │         │
         │              └──────────┘         │
         └──────────────────────────────────┘
```

一个嵌入式产品，不管多复杂，都可以拆成这四个层次来看：

| 层次 | 是什么 | 智能插座里的例子 | 成本占比（大约） |
|------|--------|----------------|----------------|
| **计算核心** | MCU（微控制器）或 MPU（微处理器） | 一个 10-15 元的 MCU 芯片 | 10-20% |
| **通信能力** | 联网模块 | WiFi 模组（8-15 元）| 10-15% |
| **传感器/执行器** | 感知物理世界 or 控制物理世界 | 电流检测芯片、继电器 | 15-25% |
| **电源与结构** | 供电电路 + 外壳 + PCB | 阻容降压电源、插座外壳 | 30-50% |

> **PM 要记住的核心概念：** 一个嵌入式产品 = 大脑（MCU） + 通信 + 感知/控制 + 供电与外壳。这四个部分互相依赖——选哪个大脑决定了你能用哪个通信模组，电源决定了传感器能跑多快。

### 1.2 MCU（微控制器）是什么，不是什么

产品经理最常见的误解是：把 MCU 当成一个"小电脑"。

| 误解 | 真相 |
|------|------|
| MCU 和电脑 CPU 差不多，只是弱一些 | 完全不同。MCU 是**单芯片集成**——CPU、内存、存储、外设接口全部印在同一块硅上。电脑的 CPU 只是一个核，还要配独立的内存条和硬盘 |
| MCU 性能越强越好 | 错。功耗、成本、封装大小、开发复杂度都会随性能剧增。选 MCU 是在四个维度上找平衡 |
| 软件可以随便往 MCU 上加 | 错。MCU 的 Flash 容量是硬上限。代码超过容量 = 物理上装不进去，不是"优化一下就能省出来" |

MCU 的几个硬指标，产品经理需要看懂：

| 参数 | 它实际上决定了什么 |
|------|------------------|
| **Flash 容量** | 你能写多少功能的代码。好比硬盘容量，单位 KB 或 MB |
| **RAM 大小** | 程序同时能处理多少数据。好比内存条，单位 KB 或 MB |
| **主频 (MHz)** | 芯片算得快不快。但嵌入式设备往往不是卡在算力，而是卡在 I/O |
| **引脚数** | 你能接多少个外设。引脚不够 = 换更大的封装（成本涨） |

> **技术细节保留：** Flash 常见 16KB ~ 2MB，RAM 常见 4KB ~ 512KB。这是芯片定价的核心分档依据。一个 Cortex-M3 内核的 STM32F103，64KB Flash + 20KB RAM，裸片价格大约 8-15 元。

### 1.3 硬件和软件在嵌入式里的特殊关系

在纯软件产品里，加功能 = 加代码。在嵌入式产品里，加功能可能意味着：

- 需要的 Flash 变大 → 换更大容量的芯片型号 → BOM 成本上升
- 需要的引脚变多 → 换更多引脚的封装 → PCB 面积变大 → 外壳可能得重新设计
- 需要的外设新类型（比如加蓝牙）→ 可能需要换芯片平台 → 软件开发推翻重来

**这是做嵌入式产品最核心的思维转变：硬件是软件的天花板，一旦定下来改了就很贵。**

## 第二章：芯片选型 —— 产品的天花板在哪里

### 2.1 第一道选择题：MCU 还是 MPU

这是最顶层的技术架构决策，决定了产品的成本基线、团队配置和开发周期。

| 维度 | MCU（微控制器） | MPU（微处理器） |
|------|----------------|----------------|
| **芯片价格** | 3 - 50 元 | 20 - 200+ 元（还要外挂 Flash/RAM） |
| **典型 BOM 成本（整板）** | 15 - 60 元 | 80 - 500 元 |
| **操作系统** | 裸机或 FreeRTOS | Linux / Android |
| **开发团队** | 2-4 名嵌入式工程师 | 需要驱动 + 系统 + 应用层工程师 |
| **原型周期** | 4-8 周 | 8-20 周 |
| **能做多复杂的事** | 传感器、控制、简单屏幕交互 | 网络协议栈、图形 UI、视频处理 |
| **典型产品** | 智能灯、温控器、电子秤、电动车窗 | 智能音箱、扫码支付终端、车载中控 |

**PM 的核心判断逻辑：**

- ✅ **选 MCU，如果：** 产品主要做采集-判断-控制循环，不需要复杂图形界面，没有需要跑第三方软件的诉求
- ✅ **选 MPU，如果：** 需要浏览器、需要复杂的 GUI 框架（Qt/Flutter）、需要运行加密协议栈或 AI 推理框架

> **经验法则：** 如果产品的交互只是几个按键 + 一个小屏幕（≤ 3.5 寸），MCU 就够了。如果涉及触摸屏滑动、网页渲染、视频播放，上 MPU。

### 2.2 MCU 内核：Cortex-M 的档位（技术细节，但 PM 需要知道）

ARM Cortex-M 是 MCU 里最主流的 CPU 内核架构，几个关键档位：

| 系列 | 算力 | 功耗 | 价格档位 | 典型应用 |
|------|------|------|---------|---------|
| **M0/M0+** | 低 | 极低（μA 级待机） | 3-8 元 | 温度传感器、电子标签、门锁 |
| **M3** | 中 | 低 | 8-20 元 | 工控、家电、物联网中端产品 |
| **M4** | 中高（带浮点运算） | 中 | 15-40 元 | 音频处理、电机控制、需要运算的产品 |
| **M7** | 高（带缓存） | 中高 | 30-80 元 | 高端传感器融合、无人机飞控 |

> **对 PM 来说：** 绝大多数 IoT 产品落在 M3 和 M4 区间。M0 适合电池续航极关键的产品。M7 通常意味着产品已经很复杂了，接近"要不要上 MPU"的边界。

### 2.3 一个现实但 PM 必须面对的问题：选国产还是进口

| | 进口（ST / NXP / Microchip） | 国产（GD / AT / CH / BL） |
|---|---|---|
| 价格 | 偏贵 | 便宜 30-60% |
| 开发工具 | 成熟（CubeIDE 等官方工具） | 通常兼容进口芯片工具链，稍有差异 |
| 文档 | 完整，英文但质量高 | 参差不齐 |
| 供货 | 2021-2023 年缺货严重，现在好转 | 近年供货稳定 |
| 技术社区 | 全球最大，资料最多 | 国内社区活跃 |
| 风险 | 地缘政治影响供应 | 少数型号可能有 IP 争议 |

**实际决策建议：** 原型阶段用进口芯片（开发快、稳定），量产时评估切换国产替代芯片的可能性。工程师说的"兼容"通常意味着"大部分兼容但有几个坑要踩"——需要留出 2-4 周的移植验证时间。

### 2.4 Flash 和 RAM：功能清单的容纳力

这是 PM 最需要看懂的硬件参数表。

| 产品类型 | 需要多少 Flash | 需要多少 RAM |
|---------|---------------|-------------|
| **极简型**（温度传感器、单按键设备） | 16-64 KB | 4-16 KB |
| **蓝牙设备 + App 交互** | 128-256 KB | 32-64 KB |
| **WiFi 连网 + 小屏显示** | 512 KB - 2 MB | 128 - 512 KB |
| **带音频/复杂图形** | 2 MB+（常需外扩 Flash） | 512 KB+ |

**现实情况：** 开发到中期发现 Flash 不够用，是嵌入式产品最常见的延期原因之一。不是工程师没写好代码，而是功能需求超过了芯片的物理容量。这时候的选择只有两个——**砍功能** 或 **换更贵的芯片**。

**产品定义阶段的黄金法则：** 按当前功能清单估算的 Flash/RAM 需求上浮 50%，作为选型基准。这个余量不是浪费，是在给产品后期的迭代买保险。

## 第三章：产品功能如何落地到硬件

### 3.1 理解"接口"和"引脚"

MCU 是一片芯片，它通过**引脚**和外面的世界连接。每个引脚可以做特定的事。一块有 48 个引脚的芯片和一块 100 个引脚的芯片，价格可能差两倍。

但好消息是：**大多数基础功能（GPIO/UART/I2C/SPI/PWM/ADC）芯片出厂就内置了，不需要额外花钱。** 真正加钱的是外接的模块——传感器、通信模组、显示屏。

### 3.2 一个功能对应什么硬件，要花多少钱

这是产品经理最需要记住的一张表：

| 你想要的功能 | 需要什么接口 | 接口是否芯片自带 | 额外 BOM 成本 |
|------------|------------|----------------|--------------|
| **按键** | GPIO | ✅ 几乎所有芯片都有 | ~0 元（仅 PCB 走线） |
| **LED 指示灯** | GPIO | ✅ 同上 | 几毛钱（LED 灯珠+限流电阻） |
| **控制继电器（开关大电流）** | GPIO + 驱动电路 | ✅ GPIO 自带，驱动电路需外加 | 2-5 元 |
| **通过手机 App 控制（蓝牙）** | UART | ✅ UART 自带 | 5-20 元（蓝牙模块） |
| **设备连 WiFi** | SPI 或 SDIO | ✅ 部分芯片自带 WiFi | 8-25 元（WiFi 模组） |
| **测量电池电压** | ADC | ✅ 大多数芯片有 | ~0 元（分压电阻） |
| **测温度/湿度** | I2C | ✅ I2C 自带 | 2-10 元（传感器芯片） |
| **驱动 OLED 屏幕** | SPI 或 I2C | ✅ 自带 | 屏幕 5-30 元 |
| **让电机转起来或调亮度** | PWM | ✅ 大多数芯片有 | 驱动芯片 + MOS 管 3-10 元 |
| **测距离（超声波/红外）** | GPIO + Timer | ✅ 自带 | 传感器模块 5-15 元 |
| **播放简单提示音** | PWM + DAC | ✅ 大多数有 | 喇叭 1-5 元 |
| **设备间长距离通信** | CAN | 部分芯片自带 CAN 控制器 | CAN 收发器 2-5 元 |

> **技术细节：** 每条通信协议都有物理层差异。UART 需要两根线（TX/RX），I2C 需要两根线（SCL/SDA）且支持多设备挂在一根总线上，SPI 需要四根线（SCK/MOSI/MISO/CS）且速度最快。这些细节在跟硬件工程师确认引脚分配时会用到。

### 3.3 现实产品案例：拆一个智能门锁的 BOM

有了上面的概念，我们可以看一个真实产品的硬件成本构成（仅供参考，量级对）：

| 部件 | 接口 | 估算成本 |
|------|------|---------|
| MCU（带低功耗蓝牙） | — | 12 元 |
| 指纹传感器 | SPI | 15 元 |
| 蓝牙天线 + 匹配电路 | 射频引脚 | 2 元 |
| 电机驱动（开锁） | PWM + GPIO | 5 元 |
| 电池管理 + 充电 | ADC 检测电压 | 3 元 |
| 按键 + LED 指示 | GPIO | 1 元 |
| RTC 时钟芯片 | I2C | 3 元 |
| 蜂鸣器 | PWM | 1 元 |
| PCB + 外壳 + 电池 | — | 20-40 元 |
| **整机 BOM 合计** | | **约 60-80 元** |

这个例子想说的是：**MCU 本身（12 元）只占整机 BOM 的 15-20%**，但 MCU 选错了，其他所有外设都得跟着变。这就是为什么芯片选型是产品定义阶段最重要的决策之一。

### 3.4 硬件的物理限制：大小和功耗

产品经理的 PRD 里经常写"体积尽量小、续航尽量长"。但这两个指标在实际中是**互相矛盾、且都有明确物理下限**的：

**关于体积：**
- MCU 封装从 3mm×3mm 到 14mm×14mm
- 天线需要特定面积（蓝牙天线至少 15mm×6mm）
- 电池体积直接决定容量
- 多层 PCB 比双层贵 2-3 倍，但能做更小的面积

**关于功耗（技术细节但很有用）：**
- MCU 工作电流：mA 级
- MCU 待机电流：μA 级（几微安）
- WiFi 发射峰值：100-300 mA（这个很耗电）
- 蓝牙 BLE 工作电流：5-15 mA
- 传感器工作电流：几十 μA 到几 mA

**快速估算：** 一块 500mAh 的电池，如果设备工作平均电流 10mA，理论续航 = 500 ÷ 10 = 50 小时 ≈ 2 天。如果要续航一年，平均电流必须小于 500 ÷ (365×24) ≈ 0.057mA = 57μA。这就是为什么长续航设备必须大部分时间处于睡眠模式。

## 第四章：软件架构决策 —— 裸机、RTOS、还是 Linux

### 4.1 为什么 PM 也要理解软件架构

软件架构的选择直接影响开发周期、功能上限和后期维护成本。它是 PM 和工程团队之间最容易被忽略的**关键对齐节点**。

### 4.2 三种方案的对比

| | 裸机 (Bare Metal) | RTOS（实时操作系统） | Linux（完整 OS） |
|---|---|---|---|
| **本质** | 一个 while 循环跑所有逻辑 | 多个任务轮换执行 | 多进程多线程 |
| **开发难度** | 低 | 中 | 高 |
| **功能上限** | 低（任务多了代码极其难维护） | 中高 | 高 |
| **内存占用** | 几乎无额外开销 | RTOS 内核 ~ 几 KB | 至少 8-16 MB |
| **复杂度临界点** | 当你要同时处理 >3 件事时 | 当实时性要求极高或后续迭代频繁时 | 当需要复杂网络/图形/多媒体时 |
| **调试难度** | 容易 | 中等 | 复杂（需要 Linux 驱动经验） |
| **量产风险** | 低 | 低 | 中（系统复杂度带来偶发 bug） |
| **典型代表** | 遥控器、电子秤、温控器 | 智能家居网关、医疗设备 | 智能音箱、车载中控 |

> **技术细节：** RTOS 不会让 MCU 变快，但它能让多个"同时要做的事情"以确定性的方式轮流执行。如果裸机单循环里一个传感器读取阻塞了 100ms，整个系统的按键响应、屏幕刷新都会一起卡。RTOS 把每个功能拆成独立任务，一个卡住不影响其他的。

### 4.3 决策树：你的产品该选哪种

```
产品需要做的功能有没有超过 3 个"同时发生的事"？
├── 没有 → 裸机就够了（开发最快，成本最低）
└── 有 → 是否需要复杂图形界面 / 网页 / 第三方软件？
    ├── 是 → 上 Linux（但需要 MPU，BOM 成本大幅上升）
    └── 否 → 上 RTOS（MCU 照用，开发周期增加 2-4 周但值得）
```

### 4.4 RTOS 里的几个关键概念（PM 需知）

| 概念 | 一句话解释 | 为什么 PM 要知道 |
|------|-----------|----------------|
| **任务/线程** | 每个 while(1) 做的事（读传感器、刷屏幕、通信） | 产品功能越多，任务越多，RTOS 的优势越明显 |
| **抢占式调度** | 高优先级的事能打断低优先级的事 | 但可能出现优先级反转——高优先级等低优先级时被中优先级插队，导致关键功能延迟 |
| **看门狗** | 硬件定时器，程序卡死了就自动重启 | 这是嵌入式设备比普通电脑更可靠的原因之一——死锁了能自己活过来 |
| **队列/信号量** | 任务之间传递数据的方式 | 任务之间的"对话"越复杂，软件调试越花时间 |
| **临界区** | 不能被打断的代码段，通常很短 | 如果工程师说要"关中断保护"，意思是有一小段代码不能被其他任务打断——正常现象，但关中断太久会影响实时性 |

> **经验之谈：** 很多团队在原型阶段图快用了裸机，做到后期功能爆炸才被迫重构成 RTOS——等于整个软件重写一遍。如果你的产品在需求文档阶段就已经规划了"App 配网 + OTA 升级 + 云平台交互 + 本地传感器 + 屏幕显示"五个功能模块，一开始就从 RTOS 起步。

### 4.5 嵌入式里的可靠性机制

| 机制 | 它做了什么 | 如果没做好会怎样 |
|------|-----------|----------------|
| **看门狗** | 硬件周期性检查程序是否还活着，死锁了自动复位 | 设备死机后不会自愈，用户只能拔电源 |
| **OTA 升级** | 通过无线网络更新固件 | 无法远程修 bug，产品上市后发现问题只能召回 |
| **双 Bank 升级** | Flash 分成两个区，升级失败能回退到旧版本 | 升级中途断电 = 变砖，只能返厂 |
| **低功耗管理模式** | 不需要时进入睡眠模式节省电量 | 续航只有预期的 1/10 |

## 第五章：从原型到量产 —— 完整开发流程

### 5.1 标准时间线（8 个阶段）

这是硬件产品特有的节奏——和纯软件产品最大的区别是：**打板、芯片采购、认证这些环节的时间是物理上不可压缩的。**

```
阶段                         时间           PM 的主要工作
────────────────────────────────────────────────────────────
需求定义 →                    W1-W2         输出 PRD，确认功能清单，框定成本目标
                                             做出关键决策：有线/无线？电池/插电？屏幕大小？
                                             
芯片选型 + 原理图 →           W3-W4         和硬件工程师确认芯片选型，下采购单
                                             注意：芯片交期可能 8-16 周，要提前订
                                             
PCB  Layout + 打板 →          W5-W6          审外观结构，确认接口位置
                                             打板费：2 层板 ~ 500 元，4 层板 ~ 2000 元
                                             
样板焊接 + 固件开发 →         W7-W12         最核心的开发阶段
                                             软件工程师在这段时间写代码、调外设
                                             通常 W8 就能看到第一个"点灯"版本
                                             
功能联调 →                    W13-W14        所有外设和通信调通
                                             这个阶段最容易发现"硬件选型失误"（比如引脚不够）
                                             
低功耗调优 →                  W15-W16        如果产品是电池供电，这个阶段非常重要
                                             不是写代码，是逐个排除谁在偷偷耗电
                                             
小批量产 + 产测 →             W17-W18        编写产线测试脚本
                                             确保每块板子出厂前跑一遍功能测试
                                             
认证 →                        W19-W22        蓝牙 BQB / WiFi FCC-SRCCE / EMC
                                             认证通常 4-8 周，费用 2-10 万/项
```

**总周期：最快 16 周，通常 22-26 周。** 如果包含复杂认证或首次使用的芯片方案，可能到 30 周以上。

**PM 最该盯的几个关键节点：**
1. **芯片采购（W4）** —— 没芯片一切白搭，交期长的一定提前下单
2. **第一版样板点亮（W8）** —— 确认电路没画错，供电正常，MCU 能跑
3. **联调完成（W14）** —— 确认所有功能理论上都能工作
4. **送样认证（W19）** —— 送出去之后就只能等了

### 5.2 BOM 成本分解（以中等复杂度产品为例）

一个带 WiFi + 小屏幕的智能家居产品，量产 10000 台级别的典型成本结构：

| 成本项 | 占比 | 说明 |
|-------|------|------|
| 芯片（MCU + 存储） | 15-20% | 通常 8-25 元 |
| 通信模组（WiFi/BLE） | 10-15% | 通常 8-20 元 |
| 传感器/执行器 | 10-20% | 差异极大，取决于产品 |
| PCB + 焊接 | 10-15% | 按面积和层数算 |
| 外壳 + 包装 | 15-25% | 模具费另计（一套模 2-10 万） |
| 电源 | 5-10% | 电池或电源适配器 |
| 认证均摊 | 2-5% | 首次认证费分摊到各批次 |
| 包装 + 配件 | 5-10% | 说明书、数据线、包装盒 |

**BOM 总成本参考：** 中等复杂度 WiFi 产品，量产 10000 台，BOM 约 50-120 元。零售价通常是 BOM 的 3-5 倍（覆盖研发、认证、渠道、售后、利润）。

### 5.3 嵌入式产品最常见的踩坑

| 踩坑 | 发生阶段 | 为什么 | 对产品的影响 | PM 能做什么 |
|------|---------|-------|------------|-----------|
| **Flash 不够** | W10-W12 | 功能超出芯片容量 | 砍功能或换芯片，延期 2-4 周 | 需求阶段就给 Flash/RAM 留 50% 余量 |
| **功耗超标** | W15-W16 | 硬件选型没考虑全系统功耗 | 续航只有目标的一半 | 立项时明确续航目标，核算理论功耗 |
| **天线性能差** | W13-W14 | 结构设计遮挡天线 | 配网失败率高 | 要求硬件在结构设计阶段就把天线位置定好 |
| **认证不通过** | W19-W22 | 电磁干扰超标 | 延期 4-8 周，重做 PCB | 预留认证预算和时间，不要在认证前改电路 |
| **选型失误** | W7-W8 | 引脚不够、外设不支持 | 需要换平台，软件重写 | 选型阶段做充分的兼容性评估 |
| **结构干涉** | W13-W14 | 外壳设计没留足空间 | 重新开模，费用 2-10 万 | 要求硬件和结构工程师提前 3D 对齐 |

### 5.4 外包 vs 自研的决策

如果你的团队没有嵌入式工程师，通常有两条路：

| | 找外包公司 | 招聘自建团队 |
|---|---|---|
| 适合阶段 | 原型验证、小批量 | 长期产品线、核心产品 |
| 成本 | 一个项目 5-30 万 | 一个嵌入式工程师月薪 15-35K |
| 风险 | 代码质量不可控，后期交接难 | 团队磨合需要时间 |
| 周期 | 通常比自研快（人家有经验） | 前期慢，但迭代后更快 |

**折中做法：** 原型用外包做，同时招一个内部的嵌入式"看门人"——不一定要自己写代码，但能看懂方案、审核技术选型、把控外包交付质量。

## 第六章：跟嵌入式团队高效沟通的实用手册

### 6.1 工程师说的这些话到底是什么意思

**"Flash 打满了"**
- 字面意思：芯片的存储容量用完了，再加任何功能都装不进去
- PM 能做的：拿出功能清单，和工程师一起砍优先级。或者同意换更大 Flash 的芯片（通常贵 2-5 元，可能会影响 PCB 兼容性）

**"这个引脚不够用了"**
- 字面意思：我们需要控制的外设数量超过了芯片的物理引脚数
- PM 能做的：接受换更多引脚的芯片（成本增加、PCB 面积变大），或者去掉一些非必要的外设

**"低功耗还没调"**
- 字面意思：芯片现在处在"全速跑"模式，没有进入睡眠。这个阶段不是写代码，是逐个关掉外设测电流
- PM 能做的：别催进度，这个是纯物理调试，基本看板和工具决定速度

**"要换大封装的型号"**
- 字面意思：同样内核的芯片，引脚更多/Flash 更大的版本。PCB 可能要做改动
- 成本影响：芯片贵 2-10 元，PCB 可能得重新 Layout，增加 1-2 周

**"这个需要上 Linux"**
- 字面意思：MCU 处理不了这么复杂的功能，得换 MPU 方案
- 成本影响：BOM 可能翻倍，开发周期延长 1-2 个月，团队需要懂 Linux 的工程师

**"在做双 Bank 升级"**
- 字面意思：把 Flash 分成两个区，升级时写 A 区用 B 区跑，写完后切换。升级失败还能退回旧版本
- PM 要知道：这是最安全的 OTA 方案，但会吃掉你大概一半的 Flash 容量。如果 Flash 本来就很紧，这功能加不进去

**"产测脚本还没写好"**
- 字面意思：还没写自动化测试的程序。每块板子出厂前需要跑一遍功能测试，烧录固件 + 验证功能
- PM 要知道：这是量产前最后一道关，别压缩这个阶段。不良品流到用户手里，退货成本远超多测一天的工时

### 6.2 给 PM 的一个"反向提问清单"

下次工程师跟你说"做不了"的时候，你可以问几个问题来判断是真不行还是条件没谈拢：

1. **"是芯片硬件不支持，还是开发时间不够？"** —— 前者需要换平台，后者加人加时间就行
2. **"换一个更贵的芯片能解决吗？大概加多少成本？"** —— 很多时候问题出在成本约束而不是技术
3. **"这个功能在其他同行产品上是做出来的，他们是怎么做的？"** —— 暗示这个功能是可行的，需要找到做法
4. **"如果功能上简化一点（比如降分辨率、延长响应时间），能做吗？"** —— 给工程师一个降级的选项而不是一刀切的"做不了"
5. **"这个风险的概率有多大？最坏情况是什么？"** —— 很多工程师说"有风险"指的是"万一出问题会很麻烦"，但实际概率很低

## 附录：快速术语表（PM 版）

保留技术定义，但每一条都附带一句"为什么 PM 需要知道"。

| 术语 | 一句话定义 | 为什么 PM 需要知道 |
|------|-----------|-------------------|
| **MCU** | 微控制器，单芯片集成 CPU+Flash+RAM+外设 | 大多数 IoT 产品的核心器件，选它还是选 MPU 决定了产品成本和复杂度上限 |
| **MPU** | 微处理器，只有 CPU 核心，需外接内存和存储 | 当产品需要跑 Linux、复杂 UI 或多媒体时被迫的选择，BOM 成本是 MCU 方案的 3-5 倍 |
| **ARM Cortex-M** | MCU 领域最主流的 CPU 内核架构（M0/M3/M4/M7） | 类似"Intel i3/i5/i7"的档次概念，越高级越贵越耗电 |
| **Flash** | 芯片内置的存储空间，放代码和固定数据 | 等同于"这个芯片能写多少功能"。Flash 满了 = 要不砍功能要不换芯片 |
| **RAM** | 芯片内置的运行内存，放临时数据 | 决定程序能同时处理多少数据。太小会导致跑不起来 |
| **GPIO** | 通用输入输出引脚，控制高低电平或读取数字信号 | 按键、LED、继电器这些最基础的功能都靠它 |
| **PWM** | 模拟电压输出的方式，控制亮度、转速、音量 | 调光、调速、调音——三种最常见的产品功能 |
| **ADC** | 把物理世界的模拟量（电压）转化成数字 | 测量电池电量、读取传感器值都需要它 |
| **UART** | 最简单的串行通信协议，连接蓝牙/WiFi/4G 模块 | 大多数通信模组都用这个接口和 MCU 对话 |
| **I2C / SPI** | 芯片与芯片之间的通信总线 | 接传感器、屏幕、存储芯片——物联网产品内部最常用的两种通信方式 |
| **CAN** | 工业/汽车领域的可靠通信总线 | 如果你的产品需要在多设备之间传输控制指令（比如无人机、车控），大概率用这个 |
| **RTOS** | 实时操作系统，保证任务在指定时间内被处理 | 当产品有多个"同时要做的事"时，需要 RTOS 来协调。开发量比裸机大，但维护成本低 |
| **裸机** | 没有操作系统，一个 while 循环跑所有逻辑 | 功能少的产品用这个最快。超过 3 个并发任务后就很难维护了 |
| **FreeRTOS** | 最流行的开源嵌入式 RTOS | ST/ESP/众多芯片厂商官方支持，生态最成熟 |
| **看门狗** | 硬件级定时器，程序卡死超时自动复位设备 | 嵌入式设备为什么比电脑"稳"的秘密——死锁了能自己恢复 |
| **OTA** | 通过无线网络远程更新固件 | 产品上市后能不能修 bug、加功能，就看有没有 OTA |
| **BOM** | 物料清单，所有元器件 + 结构件的列表和单价 | 产品定倍率和零售价的基础 |
| **PCB** | 印刷电路板，承载所有元器件和电路走线的板子 | PCB 的层数、面积直接影响整机成本。打板需要 1-2 周 |
| **原理图** | 电路连接的设计图，电气工程师的产出物 | 评审原理图时 PM 需要确认的是：接口位置、外部连接器的类型 |
| **Layout** | 把原理图变成 PCB 上实际走线布局 | Layout 完成后就不能改电路了，改一次 = 重新打板 1-2 周 |
| **EVT/DVT/PVT** | 工程验证 → 设计验证 → 产线验证，产品成熟度递增的三个阶段 | 这是硬件产品研发的里程碑，每个阶段结束前不要推进下一个阶段 |
