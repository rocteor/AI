# 硬件产品经理实战手册

> 对接开发、管理需求、原型设计 —— 从需求到落地的完整工作流

---

## 第一章：硬件 PM 的工作流全景

### 一个硬件需求从来到走，经过哪些人

```
需求方（老板/客户/市场）
    ↓
  产品经理 —— 拆解、评估、排期
    ↓
  硬件工程师 —— 芯片选型、原理图、Layout
    ↓
  嵌入式工程师 —— 固件开发、驱动调试
    ↓ (有需要时)
  App/前端工程师 —— 用户交互界面
    ↓
  结构工程师 —— 外壳、散热、装配
    ↓
  测试工程师 —— 功能测试、可靠性测试、认证
    ↓
  试产 → 量产
```

**你（PM）是这根链条里唯一从头看到尾的人。**

### 硬件 PM 和软件 PM 最大的区别

| 维度 | 软件 PM | 硬件 PM |
|------|---------|---------|
| 迭代速度 | 周级，甚至天级 | 月级到季级 |
| 改错的成本 | 修 bug 重发版本 | 改一次 PCB = 重新打板 2 周 + 几千块 |
| 并行工程 | 前后端可以独立开发 | 硬件、固件、结构必须串行协作 |
| 供应链 | 几乎没有 | 芯片交期 8-16 周，认证 4-8 周 |
| 需求变更 | 可以接受，改代码就行 | 后期变更可能意味着换芯片/重新开模 |

**核心思维转变：** 在硬件里，每个需求变更都有物理成本和时间成本。不是"这个功能加一下"，而是"加这个功能需要换更大 Flash 的芯片，成本加 5 元，开发周期延 2 周"。

## 第二章：需求来了怎么办

### 2.1 需求接收 —— 不只是记下来

当老板/客户/市场丢过来一个需求时（比如"我们要做一个智能宠物喂食器"），不要直接跟开发说。首先做一个**技术预评估**：

**第一步：拆解功能清单**

把模糊的需求拆成具体的功能项。比如"智能宠物喂食器"拆成：

```
□ 定时出粮 —— MCU + 步进电机 + RTC
□ 手机远程控制 —— WiFi/BLE 模块 + 云平台
□ 余粮检测 —— 超声波/红外传感器
□ 出粮记录查看 —— 云存储 + App 端历史数据
□ 低电量提醒 —— ADC 检测电池电压
□ 防卡粮机制 —— 电机堵转检测
```

**第二步：对每一功能项做四维评估**

| 维度 | 你要问的问题 |
|------|------------|
| 技术可行性 | 这个功能当前方案能不能做？需要新芯片吗？ |
| 成本影响 | 加这个功能 BOM 涨多少？ |
| 时间影响 | 开发这个功能需要多久？会不会影响其他功能？ |
| 风险等级 | 方案成熟吗？有现成的方案还是需要研发？ |

**第三步：输出"技术可行性摘要"**

给开发团队一份简短的需求评估文档，格式如下：

```markdown
## 需求：智能宠物喂食器 V1

### 关键功能及评估

| 功能 | 方案 | 可行性 | 成本影响 | 估算工时 | 备注 |
|------|------|--------|---------|---------|------|
| 定时出粮 | 步进电机 + RTC | ✅ 成熟方案 | +15 元 | 2w | |
| 手机控制 | ESP32 WiFi + MQTT | ✅ 有现成方案 | +20 元 | 4w | 云平台费用另计 |
| 余粮检测 | 超声波模块 HC-SR04 | ⚠️ 建议改红外 | +3 元 | 1w | 超声波在粮仓内反射不准 |
| 低电量提醒 | ADC 分压检测 | ✅ 零成本 | — | 0.5w | |
| 防卡粮 | 电机电流检测 | ⚠️ 需要额外电路 | +5 元 | 2w | 结构设计配合也很关键 |

### 总体评估
- 预估 BOM 成本：60-80 元
- 预估开发周期：12-16 周
- 主要风险点：WiFi 配网体验、防卡粮机械结构
```

### 2.2 和开发团队开会时，带什么去

**硬件评审会**和软件站会完全不同。硬件评审需要明确交付：

| 评审类型 | 参与人 | PM 要带什么 |
|---------|-------|------------|
| 立项评审 | 全员 | 功能清单 + 竞品分析 + 成本目标 + 时间线 |
| 方案评审 | 硬件 + 嵌入式 | 功能清单 + 接口需求表 + 功耗要求 |
| 原理图评审 | 硬件 | 外部接口定义 + 电源需求 + 尺寸约束 |
| 结构评审 | 硬件 + 结构 | 外观设计图 + 按键/屏幕/接口位置图 |
| 试产评审 | 全员 | 完整 BOM + 测试用例 + 认证要求 |

**每个评审会 PM 要问的三个核心问题：**
1. 当前方案有没有技术盲区？（没见过的新方案需要 POC 验证）
2. 现有的时间线里，风险最大的是哪一段？（打板？联调？认证？）
3. 如果今天要砍一个功能，砍哪个影响最小？（永远准备好 Plan B）

### 2.3 需求变更处理流程

硬件产品的需求变更必须走**正式流程**，不能口头一句"加个功能"：

```
收到变更请求
    ↓
PM 评估变更影响（技术/成本/时间/风险）
    ↓
更新需求文档 / PRD
    ↓
和开发团队确认可行性
    ↓
更新排期和 BOM
    ↓
同步给相关方（采购/生产/市场）
```

**变更常见类型和应对：**

| 变更类型 | 典型示例 | 影响 | PM 对策 |
|---------|---------|------|---------|
| 加传感器 | "能不能加个温度检测" | 可能只需加 I2C 传感器，成本低 | 如果引脚有余量，通常可行 |
| 改通信方式 | "蓝牙改 WiFi" | 可能换芯片平台，全部重做 | 成本很高，尽量在方案阶段定死 |
| 加显示屏幕 | "加个 OLED 屏" | 增加引脚占用、Flash、功耗 | 评估引脚和 Flash 余量 |
| 改外壳颜色 | "要磨砂黑" | 可能只是换料 | 如果模具已开，换颜色可能涉及新模具 |
| 改电池容量 | "续航要翻倍" | 电池体积变大，外壳可能需要改 | 结构上预留空间很重要 |

## 第三章：怎么和硬件团队高效沟通

### 3.1 团队里各角色在做什么

| 角色 | 核心工作 | 产出的东西 | PM 需要从他们那里知道什么 |
|------|---------|-----------|----------------------|
| **硬件工程师** | 芯片选型、画原理图、PCB Layout | 原理图、BOM、PCB 文件 | 芯片交期、引脚余量、BOM 成本 |
| **嵌入式工程师** | 写固件、调外设驱动 | 固件代码、烧录文件 | Flash/RAM 占用率、开发进度 |
| **结构工程师** | 设计外壳、散热、装配 | 3D 模型 CAD 文件 | 尺寸约束、模具费用、开模周期 |
| **App 工程师** | 写手机的交互界面 | App 代码、APK | 蓝牙/WiFi 通信协议、用户交互流程 |
| **测试工程师** | 功能测试、可靠性测试 | 测试报告、问题清单 | 测试覆盖率、复现率最高的 bug |

### 3.2 硬件 PRD 长什么样

软件 PRD 和硬件 PRD 的最大区别：硬件 PRD 必须包含**接口定义**和**物理约束**。

**一个硬件 PRD 的骨架：**

```markdown
# PRD：智能宠物喂食器 V1

## 1. 产品概述
- 产品定位：中端智能宠物喂食器，目标售价 299 元
- 目标用户：上班族养猫/狗人群
- 竞品参考：小米/小佩同类产品

## 2. 功能清单（按优先级）
P0（必须有）：
- 定时出粮（每天最多 4 个时段）
- 手机远程控制出粮（WiFi）
- 余粮低告警
- 防卡粮

P1（应该有）：
- 出粮记录查看（7 天历史）
- 低电量提醒

P2（最好有）：
- 语音录制播放
- 摄像头查看宠物进食

## 3. 硬件接口需求
| 接口 | 连接什么 | 数量 | 电压/协议 |
|------|---------|------|----------|
| GPIO | 步进电机驱动板 | 4 | 3.3V |
| PWM | 舵机（备用方案） | 1 | 50Hz |
| UART | WiFi 模块 | 1 | 115200bps |
| I2C | 红外传感器 | 1 | 100kHz |
| ADC | 电池电压检测 | 1 | 0-4.2V |
| GPIO | 指示灯 LED | 2 | 3.3V |

## 4. 功耗要求
- 待机功耗：< 50μA（电池供电时）
- 工作功耗：< 500mA（电机转动时）
- 电池容量：18650 一节（2600mAh）
- 目标续航：30 天（每天出粮 2 次）

## 5. 物理尺寸
- 整机尺寸：200×150×250mm
- 粮仓容量：2L
- 按键/指示灯位置：顶部面板
- 屏幕：无（V1 版本不带屏幕）

## 6. 通信协议（App ↔ 设备）
- 配网方式：蓝牙辅助配网
- 云端协议：MQTT
- 本地控制协议：UART AT 指令集
```

**给 PM 的建议：** 接口需求表是硬件 PRD 里最重要的一张表。它直接决定了：
- 芯片需要多少个引脚（决定了芯片价格和封装）
- PCB 上需要布多少线（决定了 PCB 尺寸和层数）
- 外壳上需要开多少个孔（决定了模具复杂度）

### 3.3 正确问问题的方式

PM 问问题的措辞决定了工程师会怎么回答。对比两组对话：

**❌ 低效版：**
```
PM："这个功能能不能做？"
工程师："做不了。"
PM："为什么做不了？"
工程师："这个芯片不支持。"
PM："那换一个芯片呢？"
工程师："要评估一下。"
—— 对话死了
```

**✅ 高效版：**
```
PM："这个功能需要 MCU 支持 I2S 音频接口，我们现在用的芯片有吗？"
工程师："STM32F103 没有 I2S，要换到 F4 系列才有。"
PM："F4 系列的封装和 F103 兼容吗？成本差多少？"
   —— 对话变成了具体的技术决策
```

**几个 PM 可以用的典型追问：**

1. *"这个是芯片硬件不支持，还是开发时间不够？"* —— 前者要换芯片，后者加人加时间
2. *"如果功能上降级（比如降低采样率/减少存储天数/延长响应时间），能做吗？"* —— 给出降级选项
3. *"这个风险的触发概率有多大？最坏情况是什么？"* —— 区分"有可能出问题"和"大概率会出问题"
4. *"如果成本预算加 10 元，能解决吗？"* —— 很多时候问题出在成本约束而不是技术可行性

## 第四章：用 Figma 做交互原型

### 4.1 Figma 基础 —— 15 分钟上手

Figma 是一款基于浏览器的 UI 设计工具。不需要安装（但有桌面版），免费版足够个人使用。

**核心概念（用开发的语言说）：**

| Figma 术语 | 对应开发概念 |
|-----------|------------|
| Frame | 相当于一个 div/view/screen，定义画布尺寸 |
| Layer | 相当于 HTML 元素，一个 Frame 里可以套无数层 |
| Component | 相当于可复用的 React 组件/Vue 组件 |
| Auto Layout | 相当于 flexbox，自动排列子元素 |
| Constraints | 相当于相对定位/响应式布局 |
| Prototype | 交互连线，相当于路由跳转 |
| Variants | 同一个组件的不同状态（hover/pressed/disabled） |

**第一步：创建一个 Frame**
1. 打开 Figma，点 + 号 → 选手机模板（iPhone 14 / 15）
2. 这就是你的第一个屏幕画布

**第二步：拖拽基础元素**
- 用快捷键 R 画矩形（按钮）
- 用快捷键 T 打字（标签）
- 用快捷键 O 画圆形（图标背景）
- 从右侧面板调整颜色、圆角、阴影

**第三步：用 Auto Layout**
选中一个 Frame，按 Shift+A 开启 Auto Layout。它的作用：
- 子元素自动水平或垂直排列
- 间距统一控制
- 内容增减时自动撑开

**这是 Figma 里最重要、最值得花 30 分钟练习的一个功能。**

### 4.2 硬件产品的交互设计特点

硬件产品的 App 交互和纯 App 不一样：

| 维度 | 纯 App 设计 | 硬件配套 App 设计 |
|------|-----------|-----------------|
| 页面数量 | 可能几十上百个页面 | 通常 5-15 个（配置/控制/设置/关于） |
| 核心交互 | 滑动、点击、长按 | 开关、滑块、定时、状态展示 |
| 设备连接 | 无 | 配网/绑定/OTA 升级是核心流程 |
| 离线状态 | 不常见 | 必须考虑设备离线、断连、低电量 |
| 错误处理 | 弹窗提示 | 要考虑设备无响应/指令超时/固件更新失败 |

**硬件 App 的典型页面结构：**

```
引导页（首次使用）
    ↓
设备配网（WiFi 配网 / 蓝牙绑定）
    ↓
主控制页（开关/调节/模式切换）
    ↓
设置页（定时/通知/固件升级/关于）
    ↓
设备列表（如果有多个设备）
```

### 4.3 从线框图到高保真原型

**第 1 步：手绘线框图（纸上）**

不要一开始就打开 Figma。先用笔和纸画出每个页面的布局。关注：
- 信息的优先级（什么功能放在最显眼的位置）
- 操作路径（用户点完第一步之后去哪里）
- 物理设备的对应关系（屏幕上的开关和真实设备的按键对应吗）

**第 2 步：Figma 低保证线框图（灰度）**

用灰度方块代替所有 UI 元素。不配色、不选字体、不用真实图片。目的是验证布局和交互流程。

**第 3 步：Figma 高保真原型**

加上真实配色、字体、图标、图片。用 Prototype 模式连接页面之间的跳转。

**给硬件 PM 的省钱建议：** 如果你的团队没有专职设计师，这套流程里做到第 2 步（灰度线框图）就够了。灰度线框图足够给开发讲清楚"这个页面长什么样、按了这个按钮去哪"。真正的高保真可以等产品验证通过后再交给设计师精修。

### 4.4 Figma 的组件化思维

做硬件 App 时，以下组件值得你先做好然后复用：

```markdown
通用组件库（建议第一周就建好）：
├── 设备卡片（设备名、状态、信号强度）
├── 开关控件（开/关/正在切换中）
├── 滑块（亮度/音量/温度调节）
├── 定时卡片（时段起始/重复/开关）
├── 状态标签（在线/离线/升级中/故障）
├── 配网流程页（搜索设备 → 连接 → 配网成功/失败）
├── 底部导航栏
└── 加载/空状态/错误状态
```

在 Figma 里把这些做成 Components（Component 创建后，你改一个实例，所有实例同步更新），后续拼接页面就像搭积木。

## 第五章：用 AI 生成 App 设计草稿

### 5.1 为什么硬件 PM 应该用 AI 做设计

你不需要成为设计师，但你需要能快速产出"够用的交互原型"给开发参考。AI 可以帮你：
1. 从文字描述直接生成 UI 页面
2. 提供多种设计方案供选择
3. 快速产出设计规范（配色、字体、间距）

### 5.2 AI 设计工具推荐

| 工具 | 适合场景 | 费用 | 产出格式 |
|------|---------|------|---------|
| **UIzard** | 文字 → UI，自动生成可用 Figma 文件 | 免费有额度 | Figma 文件 |
| **Galileo AI** | 文字描述生成高保真 UI | 付费（试用期免费） | Figma 文件 |
| **Midjourney / DALL-E** | 生成设计概念图/灵感参考 | 付费 | 图片 |
| **Claude / GPT-4** | 生成设计规范、交互流程文案 | 有额度限制 | 文字+ASCII 图 |

### 5.3 实战：用 AI 生成一个硬件 App 的 UI

**场景：** 我们要做一个"智能空气净化器"的配套 App。

**第一步：写 Prompt**

AI 生成 UI 的质量 90% 取决于 Prompt 的质量。一个好的 Prompt 格式：

```
设计一个智能设备的配套 App 页面：

设备：智能空气净化器
目标用户：家庭用户，25-45 岁，关注空气质量
核心功能：
1. 实时空气质量显示（PM2.5 / 温度 / 湿度）
2. 开关控制 + 风速调节（低/中/高/自动）
3. 定时开关机
4. 滤芯寿命提醒

设计风格：简洁、干净、信息优先
主要色彩：白底 + 蓝色点缀
需要展示的页面：主控制页

请输出页面布局的描述，包括：
- 每个区域放什么信息
- 信息的优先级顺序
- 交互方式和按钮位置
```

**第二步：从 AI 的回复中提取设计信息**

AI 会回复一段页面布局描述。提取关键信息后，在 Figma 里实现：

```
主控制页布局（从上到下）：
┌─────────────────────────┐
│  状态栏：在线 | WiFi 强  │
│  ┌─────────────────┐    │
│  │ PM2.5   35 μg/m³│    │  ← 核心数据，最
│  │ 温度 25°C 湿度 45% │    │     大字号显示
│  │ 评级: 优          │    │
│  └─────────────────┘    │
│                         │
│  ┌─ 开关 ──────────┐   │  ← 主操作按钮
│  └─────────────────┘   │
│                         │
│  风速: ○低 ○中 ○高 ●自动│  ← 模式选择
│                         │
│  定时: 未设置 [设置]    │  ← 次要功能
│  滤芯寿命: 85% [详情]   │
│                         │
│  [底部导航: 首页/设备/我的]│
└─────────────────────────┘
```

**第三步：将 AI 输出转化成 Figma 线框图**

1. 在 Figma 创建手机 Frame（iPhone 15）
2. 按 AI 给出的布局从上到下排布
3. 用 Auto Layout 控制间距

一个好的 Prompt 加上 30 分钟 Figma 操作，可以产出开发可以直接使用的线框图。

### 5.4 常用 Prompt 模板

**生成页面布局：**

## 第六章：实战案例 —— 从零设计一个智能硬件产品

### 场景

你是 PM，公司决定做一个"智能胶囊咖啡机"（对标 Nespresso 但可以手机控制）。从需求到开发交付，你该怎么走？

### 第 1 周：需求定义

**输出：** 一页纸的功能清单 + 竞品分析

```
P0（必须有）：
- 手机控制萃取（浓度/温度/杯量）
- 胶囊识别（NFC 读取胶囊信息）
- 水箱水位检测
- 废水盘满提醒

P1（应该有）：
- 定时预约
- 个人口味配置文件（3 个预设）
- 缺水/缺胶囊自动告警

P2（最好有）：
- 咖啡豆余量统计
- 冲煮历史记录
```

### 第 2 周：技术预评估 + 选型

**输出：** 技术可行性评估表

你和硬件工程师开会对了一遍：

| 功能 | 方案 | 引脚 | 评估 |
|------|------|------|------|
| NFC 胶囊识别 | I2C NFC 模块 | +2 引脚 | ✅ 成熟方案 |
| 水泵控制 | MOSFET + PWM | +2 引脚 | ✅ |
| 水位检测 | 电极式 + ADC | +1 引脚 | ⚠️ 干烧保护要单独做 |
| 加热控制 | SSR + 温度传感器 | +2 引脚 | ✅ NTC 温度检测 |
| 手机控制 | ESP32（自带 WiFi/BT） | UART | ✅ ESP32 自带双模 |

结论：一片 ESP32 能覆盖所有功能，不需要 MCU + 独立 WiFi 模组。

### 第 3 周：交互原型

**输出：** Figma 灰度线框图（5 个核心页面）

```
1. 配网引导页 —— 首次使用流程
2. 主控制页 —— 选择口味/杯量/浓度
3. 预约定时页 —— 设置冲煮时间
4. 设备状态页 —— 水箱/胶囊/废水盘状态
5. 个人设置页 —— 口味预设
```

用 AI 生成初始布局参考 → 在 Figma 调整成可交付的线框图。

### 第 4 周：PRD + 评审

**输出：** 完整 PRD

包含：
- 功能清单（P0/P1/P2）
- 接口需求表（哪个外设接哪个引脚）
- 功耗要求（目标：待机 < 1W）
- 物理尺寸约束
- App 交互流程图
- 认证要求（3C / FCC / ETL）

### 第 5 周起：进入开发

你的角色从"定义"切换到"跟进"：

- 每周和硬件工程师检查原理图进度
- 每两周看一次固件 Demo
- 跟踪芯片采购交期
- 提前联系认证机构
- 准备产测方案

### 原理图评审

- [ ] 芯片供电电压是否正确
- [ ] 所有外设的引脚没有冲突
- [ ] 预留了 SWD 调试接口

- [ ] 散热孔/风扇位置
- [ ] 防水等级要求（IPX 多少）
- [ ] 开模可行性
