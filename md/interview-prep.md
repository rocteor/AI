# 嵌入式面试与求职指南

> 写给软件转行者的面试准备手册 —— 从简历到技术面到谈薪，全覆盖。

## 第一章：转行面试的心态准备

### 1.1 你的软件背景是资产，不是负债

很多转行的人最大的心理障碍是"我不够懂硬件"。但事实上，**嵌入式开发本质上仍然是软件开发**——它只是有额外的硬件约束。

你已有的优势：

| 你有的 | 面试官看重什么 |
|--------|-------------|
| C/C++ 扎实 | 嵌入式 80% 以上的代码是 C |
| 指针和内存模型理解深 | 嵌入式里每天都在跟指针/地址/内存打交道 |
| 数据结构与算法 | 嵌入式面试同样考链表/队列/树 |
| 面向对象编程 | 嵌入式固件里的模块化设计 |
| Debug 经验 | GDB 等调试工具的使用经验可以直接平移 |
| 版本控制（Git）| 嵌入式团队同样用 Git |
| 计算机体系结构基础 | CPU 流水线、缓存、MMU——这些知识直接相关 |

**面试时的策略：** 当面试官问到一个你不清楚的硬件细节时，不要说"这个我还没接触过"，而是说"这个部分我在转行过程中正在重点补充，我理解的基本原理是……"

### 1.2 嵌入式岗位的常见分类

| 岗位 | 核心技能 | 适合人群 |
|------|---------|---------|
| 嵌入式软件工程师 | C/RTOS/驱动/外设 | 纯软件背景，最直接的对口岗位 |
| 嵌入式硬件工程师 | 电路设计/原理图/Layout | 需要 EE 背景 |
| 嵌入式测试工程师 | 测试/脚本/自动化 | 转行的过渡岗位 |
| 固件工程师（Firmware） | 汇编/C/芯片底层 | 偏底层，有汇编经验更好 |
| BSP/驱动工程师 | Linux 驱动/内核 | 有 Linux 经验的人 |
| IoT 工程师 | 无线通信/低功耗 | 蓝牙/WiFi/Zigbee 经验 |
| 硬件产品经理 | 技术理解/项目管理 | 你的目标岗位 |

**对转行者来说，嵌入式软件工程师 > 固件工程师 > IoT 工程师 > BSP 工程师 > 硬件工程师。**

## 第三章：高频面试题（含答案要点）

### 3.1 C 语言基础类

**Q: volatile 关键字的作用？**

> 告诉编译器这个变量的值可能被意想不到地改变（被硬件寄存器或 ISR 修改）。编译器不会对它做优化——每次读写都从内存地址读取，而不是使用寄存器中缓存的副本。

典型应用场景：1) 硬件寄存器地址；2) ISR 中修改的全局变量；3) 多线程共享变量。

**Q: const 关键字在嵌入式中的用法？**

1. `const int *p` — 指针指向的值是常量（指针本身可以变）
2. `int * const p` — 指针本身是常量（指向的值可以变）
3. `const int * const p` — 指针和指向的值都是常量

嵌入式应用：用 `const` 定义放在 Flash（只读存储区）中的查表数据，不占 RAM。

**Q: static 关键字的作用？**

1. 函数内的 static 变量 — 只初始化一次，函数退出后值保留
2. 文件内的 static 函数/变量 — 作用域限于本文件（C 语言级别的"私有"）
3. 文件内的 static 全局变量 — 其他文件看不到

嵌入式应用：用 static 限制变量作用域，避免全局命名冲突——这在大型嵌入式项目中尤为重要。

**Q: 什么是段错误（Segmentation Fault）？在嵌入式里对应的现象是什么？**

在桌面 C 程序里是"段错误"。在嵌入式里就是 **HardFault**。触发原因类似：访问了非法的内存地址、空指针解引用、栈溢出等。

在嵌入式里调试 HardFault 的方法：进入 HardFault_Handler 断点，查看寄存器（LR、MSP/PSP、PC），通过 PC 值反查是哪条指令触发的。

### 3.2 中断与实时性

**Q: 中断和轮询的区别？各在什么场景使用？**

| | 中断 | 轮询 |
|------|------|------|
| CPU 占用 | 事件发生时才处理，CPU 利用率高 | 持续检查，浪费 CPU |
| 响应延迟 | 微秒级 | 取决于轮询频率 |
| 复杂度 | 需要处理重入和优先级 | 简单 |
| 适合场景 | 低频事件、对延迟敏感 | 高频、可预测的查询（如 ADC 连续采样）|

**Q: 中断服务函数（ISR）有哪些限制？**

1. 必须短——ISR 执行时间越短越好，不要在 ISR 里做复杂计算
2. 不能调用可能阻塞的函数（如 HAL_Delay、printf、malloc）
3. 不能调用可能触发同一优先级或更低优先级中断的函数（避免递归）
4. 必须清中断标志——否则会反复进入
5. 数据传递用 volatile 全局变量或信号量/队列

**Q: 什么是临界区？怎么保护？**

临界区 = 不能被中断打断的代码段。用"关中断"来保护。

```c
// 进入临界区
__disable_irq();
// ... 操作共享数据 ...
__enable_irq();   // 恢复中断
```

注意：临界区时间必须极短（微秒级），否则会影响系统实时性。

### 3.3 内存与存储

**Q: MCU 的 Flash 和 RAM 分别用来存什么？**

| 存储 | 存什么 | 特点 |
|------|-------|------|
| Flash | 代码（.text）、常量（.rodata）、初始化后的全局变量（.data 的初始值）| 掉电不丢、只读（运行时）|
| RAM | 栈（局部变量、函数调用）、堆（malloc）、全局变量（.data 的运行时副本 + .bss）| 掉电丢失、读写 |

**Q: 嵌入式里为什么不建议用 malloc？**

1. 不确定的执行时间 —— malloc 的执行时间不可预测（可能触发碎片整理）
2. 内存碎片 —— 长时间运行后，碎片化可能导致分配失败
3. 失败难以处理 —— MCU 没有 swap，malloc 失败 = 系统崩溃
4. 线程不安全 —— 在 freertos 环境下，malloc 不是线程安全的

**替代方案：** 静态分配（全局数组）、定长内存池（FreeRTOS 的 `pvPortMalloc` 内部实现）。

**Q: .data 段和 .bss 段的区别？**

| | .data | .bss |
|------|-------|------|
| 存什么 | 已初始化的全局变量和 static 变量 | 未初始化的全局变量和 static 变量 |
| 初始值 | 编译时确定，存储在 Flash 中 | 启动时清零 |
| 启动过程 | 从 Flash 复制到 RAM | RAM 中直接清零 |
| 占用 Flash | ✅ 是（存初始值）| ❌ 否（不存初始值）|

### 3.4 通信协议

**Q: UART、I2C、SPI 三者的区别？**

| | UART | I2C | SPI |
|------|------|-----|-----|
| 线数 | 2（TX/RX）| 2（SCL/SDA）| 4（SCK/MOSI/MISO/CS）|
| 同步/异步 | 异步 | 同步 | 同步 |
| 全双工 | ✅ | ❌ 半双工 | ✅ |
| 多设备 | ❌ 点对点 | ✅ 总线 | ✅ 片选 |
| 速度 | ~115.2kbps - 几 Mbps | ~100kHz - 3.4MHz | ~几十 MHz |
| 硬件复杂度 | 低 | 中 | 低 |

**Q: I2C 为什么需要上拉电阻？**

I2C 是开漏总线——设备只能拉低 SDA/SCL，不能主动拉高。高电平由上拉电阻提供。上拉电阻的阻值影响：
- 太大：上升沿缓，高频通信时波形跟不上
- 太小：设备拉低时灌入电流太大
- 标准值：4.7kΩ（400kHz 以下）、2.2kΩ（1MHz+）

**Q: SPI 的四种模式（CPOL/CPHA）怎么理解？**

- CPOL = 时钟极性（空闲时 SCK 的电平）：0 = 低电平空闲
- CPHA = 时钟相位（数据在哪个边沿采样）：0 = 第一个边沿采样

两种组合：Mode 0（CPOL=0, CPHA=0）和 Mode 3（CPOL=1, CPHA=1）最常用。SSD1306 OLED 需要 Mode 3。

### 3.5 RTOS

**Q: FreeRTOS 的任务有哪几种状态？**

1. **运行中**（Running）—— 当前正在被 CPU 执行的任务
2. **就绪**（Ready）—— 可以运行但当前没有 CPU 时间
3. **阻塞**（Blocked）—— 等待某个事件（延时到期、信号量/队列可用）
4. **挂起**（Suspended）—— vTaskSuspend 挂起，只有调用 vTaskResume 才能恢复

**Q: 什么是优先级反转？怎么解决？**

**场景：** 高优先级任务 H 等待低优先级任务 L 持有的互斥量。此时中等优先级任务 M 运行（不涉及互斥量），H 被 M 间接阻塞。

**解决方案：** 优先级继承。当 H 等待 L 持有的互斥量时，L 临时继承 H 的优先级。L 释放互斥量后恢复原优先级。FreeRTOS 的互斥量（`xSemaphoreCreateMutex`）内置优先级继承机制。

**Q: 看门狗（Watchdog）的原理和正确用法？**

独立看门狗（IWDG）使用独立 RC 振荡器（LSI，约 40kHz），独立于主时钟。启动后计数器递减，到 0 时复位 MCU。

**正确用法：**
- 在主循环（或最高优先级的监控任务）中喂狗
- 不要在 ISR 中喂狗（主循环卡死但中断还在运行，看门狗不会触发）
- 喂狗间隔 < 看门狗超时时间的 50%（留余量）

### 3.6 启动流程

**Q: MCU 上电后到 main() 之间发生了什么？**

```
1. 从 0x00000000 读取栈指针（MSP）初始值
2. 从 0x00000004 读取复位向量（Reset_Handler 地址）
3. 跳转到 Reset_Handler
4. Reset_Handler 调用 SystemInit（配置时钟）
5. 调用 __main（C 运行时初始化）：
   a. 复制 .data 段从 Flash 到 RAM
   b. 清零 .bss 段
   c. 初始化堆（__heap_base, __heap_limit）
6. 跳转到 main()
```

**Q: 中断向量表放在哪里？怎么重定向？**

默认在 Flash 起始地址（0x08000000 在 STM32 上）。可以通过 SCB->VTOR 寄存器重定位到 RAM 中（用于 IAP/OTA 升级，运行期间修改中断向量）。

## 第四章：技术面之后 —— 项目和综合能力

### 4.1 准备好"做过什么"的故事

面试官问项目经验时，他们想听的不是"你做了什么"，而是**为什么这么做、考虑过哪些方案、遇到什么困难、怎么解决的**。

**好的回答结构：**

> 我在做一个温湿度监测项目时，一开始用的 DHT11 单总线传感器，因为便宜。但发现 DHT11 的采样周期太长（2 秒一次），而且精度只有 ±2°C。后来换成了 SHT30（I2C 接口，精度 ±0.3°C，采样 100ms）。
>
> 但在换传感器的过程中遇到了 I2C 通信间歇性失败的问题。用逻辑分析仪抓波形发现 SCL 和 SDA 的上拉电阻是 10kΩ，对于 400kHz 的 I2C 速度来说太大了。换成 4.7kΩ 后问题解决。
>
> 从这个项目我学会了两件事：1) 选传感器要在成本和性能之间做实测对比，不能只看参数表；2) I2C 的硬件设计细节（上拉电阻选值）会影响通信可靠性。

**这个回答展示了什么？**
- ✅ 技术判断力（知道 DHT11 和 SHT30 的区别）
- ✅ 动手能力（换传感器、抓波形）
- ✅ 解决真实问题的能力（定位到上拉电阻问题）
- ✅ 从经验中总结的能力

### 4.2 准备好"开放式问题"

**Q: 如果让你设计一个智能灯泡，你会怎么选芯片？**

**好回答的框架：**
1. 先问"目标售价多少？"——框定 BOM 预算
2. 再问"需要什么功能？"——纯开关、调色温、还是 RGB？
3. 再问"什么通信方式？"——WiFi、蓝牙、还是 Zigbee？
4. 给出具体方案：选 ESP32-C3（单芯片 WiFi/BLE / 足够 GPIO / ~15 元）
5. 提到可能的妥协：如果用 ESP32-C3 的 GPIO 驱动 PWM 调光，需要确认它是否有足够分辨率的 PWM 定时器

**Q: 我们（公司）的产品出货后，用户在寒冷地区发现电池续航大幅下降，你怎么排查？**

**好回答流程：**
1. 确认温度对电池的影响——锂电池在 -10°C 时容量会下降 30-50%
2. 确认是否只是电池问题——还是 MCU 在低温下启动了不同的工作模式？
3. 建议用高低温箱复现——在 -20°C 下测量整机功耗
4. 对策方向：换低温性能更好的电池 / 增加保温 / 修正电量计算算法

### 4.3 准备好"技术深度问题"

**Q: 你用什么工具调过 I2C 通信问题？**

如果你说"逻辑分析仪"，面试官会追问：
> 你当时看到什么波形？怎么判断是设备没响应？ACK/NACK 在波形上长什么样？

**回答：**
> 看到 SCL 在走时钟，SDA 上发送了设备地址后，在第 9 个时钟的 ACK 位上 SDA 没有被设备拉低，而是保持高电平——这说明设备没响应（NACK）。可能是设备地址错了、设备没供电、或者 I2C 总线被拉死了。

## 第五章：常见笔试/上机题

### 5.1 C 语言

**实现一个环形缓冲区（FIFO）：**

```c
typedef struct {
    uint8_t *buffer;
    int head;   // 写位置
    int tail;   // 读位置
    int size;   // 缓冲区大小（必须是 2 的幂）
} ring_buffer_t;

int rb_init(ring_buffer_t *rb, uint8_t *buf, int size) {
    rb->buffer = buf;
    rb->size   = size;
    rb->head   = 0;
    rb->tail   = 0;
    return 0;
}

int rb_push(ring_buffer_t *rb, uint8_t data) {
    int next = (rb->head + 1) & (rb->size - 1);  // 用位运算取模
    if (next == rb->tail) return -1;  // 满了
    rb->buffer[rb->head] = data;
    rb->head = next;
    return 0;
}

int rb_pop(ring_buffer_t *rb, uint8_t *data) {
    if (rb->tail == rb->head) return -1;  // 空了
    *data = rb->buffer[rb->tail];
    rb->tail = (rb->tail + 1) & (rb->size - 1);
    return 0;
}
```

**面试官会追问的：** 如果 size 不是 2 的幂，你的取模运算还能用位运算简化吗？不能的话怎么改？

### 5.2 嵌入式场景题

**让一个 LED 以 500ms 交替亮灭，不使用 HAL_Delay（只能使用硬件定时器）：**

```c
// 定时器 ISR，每 500ms 触发一次
void TIM2_IRQHandler(void) {
    if (TIM_GetITStatus(TIM2, TIM_IT_Update)) {
        TIM_ClearITPendingBit(TIM2, TIM_IT_Update);
        GPIO_ToggleBits(GPIOA, GPIO_Pin_5);
    }
}
```

**面试官会追问的：** 如果我现在要同时让另一个 LED 以 300ms 闪烁，怎么改？用同一个定时器还是另一个？

## 第六章：面试后的跟进

### 6.1 面试结束时的好问题

面试官通常会问"你有什么想问的？"。不问问题 = 缺乏兴趣。问"公司有什么福利" = 浪费机会。

**建议问的问题：**

| 问谁 | 推荐问题 |
|------|---------|
| 技术面 | "你们团队现在用的 MCU 平台是什么？正在用的 RTOS 是哪个？" |
| 技术面 | "一个典型的开发周期从需求到量产要多长时间？" |
| 技术面 | "你们遇到过最棘手的硬件问题是什么？怎么解决的？" |
| 主管面 | "这个岗位的目标岗位在前 6 个月的关键成果会是什么？" |
| 主管面 | "团队的硬件开发和软件开发是怎么协调的？" |
| HR 面 | "转行的同事进来后，团队一般会给多长时间的学习期？" |

### 6.2 转行求职的可行路径

如果直接投嵌入式岗位没有回应，可以考虑"迂回路线"：

```
路径 1：嵌入式测试 → 嵌入式开发
路径 2：技术支持/FAE → 嵌入式开发
路径 3：先做内部工具/脚本 → 介入嵌入式项目
路径 4：（已有 PM 经验）→ 直接投硬功 PM
```

**对你有 C/C++ 底子的人来说，最快的路可能是路径 4** —— 直接投硬件 PM 或 IoT PM。你的软件背景反而是一个区分度——能够和嵌入式团队用同一套语言沟通的 PM 是稀缺资源。

## 附录 B：面试中常说错的话

> **最后的话：** 转行的路上最大的障碍不是技术差距，是"我能不能行"的自我怀疑。嵌入式领域有大量从应用层转过来的工程师——因为底层对 C 语言和系统理解的要求反而让有经验的软件开发者更有优势。把你的编程思维平移过来，补上硬件感知，你就是一个有竞争力的候选人。
