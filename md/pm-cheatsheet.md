# 硬件 PM 速查手册

> 从 6 篇文档中提取的精华，一页纸搞定硬件 PM 日常工作需要。

---

## 一、三张核心速查表

### 表 1：MCU vs MPU —— 选哪个

| 你的产品需要 | 选 MCU | 选 MPU |
|-------------|--------|--------|
| 简单传感器控制 | ✅ 5-30 元 | ❌ 杀鸡用牛刀 |
| 手机 App 远程控制 | ✅ ESP32 单芯片搞定 | ❌ 太贵 |
| 彩色触摸屏交互 | ⚠️ 小屏可以，大屏吃力 | ✅ 需要 Linux 跑 GUI |
| 视频/音频处理 | ❌ 算力不够 | ✅ 必须上 Linux |
| BOM 预算 < 60 元 | ✅ | ❌ 不可能 |
| 开发周期 < 3 个月 | ✅ | ❌ 至少 4-6 月 |
| 需要 OTA 升级 | ✅ 可以 | ✅ 更成熟 |

### 表 2：Flash/RAM —— 功能够不够装

| 产品类型 | 需要 Flash | 需要 RAM | 芯片价格档 |
|---------|-----------|---------|-----------|
| 温度传感器、单按键设备 | 16-64 KB | 4-16 KB | 3-8 元 |
| 蓝牙 + App 交互 | 128-256 KB | 32-64 KB | 8-15 元 |
| WiFi + 小屏幕显示 | 512 KB - 2 MB | 128-512 KB | 15-30 元 |
| 带音频/复杂图形 | 2 MB+ | 512 KB+ | 40-80 元 |

> **黄金法则：** 按当前功能估算的 Flash/RAM 上浮 50% 作为选型基准。不是浪费，是给迭代买保险。

### 表 3：裸机 vs RTOS vs Linux —— 软件架构选哪个

| 问自己一个问题 | 答案 |
|---------------|------|
| 产品有没有超过 3 个"同时做的事"？ | 没有 → **裸机**。有 → 往下看 |
| 需要复杂 GUI / 网页 / 第三方软件吗？ | 是 → **Linux（MPU）**。否 → **RTOS（MCU）** |

**裸机：** 开发最快，但超过 3 个并发任务后代码极难维护
**RTOS：** 初期慢 2-4 周，但后期扩展性好。推荐一开始就用
**Linux：** BOM 成本翻倍，开发周期翻倍，但能做最复杂的产品

## 二、产品 Timeline —— 每站 PM 做什么

```
W1-W2   需求定义         → 出 PRD，确认功能清单，定目标成本
W3-W4   芯片选型+原理图   → 下采购单（注意芯片交期 8-16 周！）
W5-W6   PCB Layout+打板  → 打板费 2 层 ~500 元，4 层 ~2000 元
W7-W12  样板焊接+固件开发 → 最核心阶段，PM 每周看进度
W13-W14 功能联调          → 最容易发现"选型失误"
W15-W16 低功耗调优        → 电池供电产品别压缩这阶段
W17-W18 小批量+产测       → 准备产线测试，每块板子都要测
W19-W22 认证              → 蓝牙/WiFi/FCC/CE，4-8 周，2-10 万/项
```

**总周期：最快 16 周，通常 22-26 周。** PM 最该盯的两个节点：
1. **W4 芯片采购** —— 没芯片一切白搭
2. **W8 第一版点亮** —— 确认供电正常、MCU 能跑

## 三、BOM 成本快速估算

### 各功能模块成本参考（10000 台量级）

| 功能 | 最低成本方案 | 成本 |
|------|------------|------|
| MCU（低端） | STM32F030 / GD32F103 | 3-8 元 |
| MCU（中端） | STM32F103 / ESP32 | 8-20 元 |
| WiFi 模块 | ESP8266 / ESP32-C3 | 8-15 元 |
| 蓝牙模块 | HC-08 / TLSR8253 | 5-15 元 |
| 温湿度传感器 | SHT30 / DHT22 | 3-8 元 |
| OLED 屏幕 0.96" | SSD1306 | 8-15 元 |
| 指纹模组 | 电容式 | 20-30 元 |
| 继电器 | 5A / 10A | 2-5 元 |
| PCB（2 层） | 30×30mm | 2-5 元 |
| PCB（4 层） | 50×50mm | 8-15 元 |
| 外壳（注塑） | ABS 塑料 | 5-15 元（+开模费 2-5 万）|
| 电池 | 18650 一节 | 8-15 元 |
| 电源适配器 | 5V/1A | 8-15 元 |

### 整机 BOM 估算公式

```
简单产品（插座/灯/开关）:   BOM ≈ 15-30 元 → 零售 ≈ 50-100 元
中等产品（门锁/净化器）:    BOM ≈ 100-300 元 → 零售 ≈ 300-1000 元
复杂产品（摄像头/音箱）:    BOM ≈ 120-500 元 → 零售 ≈ 200-1500 元
```

**零售价 ≈ BOM × 3~5 倍**（覆盖研发、认证、开模、渠道、售后、利润）

## 四、工程师黑话翻译

| 工程师说 | 翻译成 PM 能懂的话 | PM 应该回应 |
|---------|-----------------|-----------|
| "Flash 打满了" | 芯片存储满了，再加功能物理上装不进去 | 拿出功能清单，决定砍什么或换更贵的芯片 |
| "引脚不够了" | 外设数量超过了芯片的物理引脚 | 接受换更多引脚的芯片(+2-10元)，或去掉部分外设 |
| "要换大封装" | 同内核芯片，引脚/Flash 更大的版本 | 成本增加 2-10 元，PCB 可能重做，延 1-2 周 |
| "低功耗还没调" | 芯片还在全速跑，没进睡眠模式 | 这是物理调试，催也没用，别压缩这阶段时间 |
| "这个需要上 Linux" | MCU 处理不了，得换 MPU 方案 | BOM 翻倍，周期延长 1-2 月 |
| "在做双 Bank 升级" | Flash 分两个区，升级失败能回滚 | 安全的 OTA 方案，但会吃掉一半 Flash |
| "信号有毛刺" | 波形有噪声，可能误触发 | 硬件问题，可能需要改善 PCB Layout |
| "电源纹波太大" | 电源输出不干净，影响传感器精度 | 可能是 DCDC 输出缺滤波电容 |
| "地弹" | 大电流瞬间抬起地线电压 | 大功率设备（电机）容易出现 |

## 五、需求评估 Checklist

拿到一个新需求，按这个顺序问一遍：

### 技术可行性
- [ ] 这个功能现有芯片方案能支持吗？
- [ ] 需要换芯片平台吗？换了之后现有软件要重写吗？
- [ ] 这个方案有人做过吗？有参考设计吗？

### 成本影响
- [ ] 加这个功能 BOM 涨多少？
- [ ] 需要新的模具/治具吗？（开模费多少？）
- [ ] 认证费用会变吗？（加了 WiFi/蓝牙需要额外认证）

### 时间影响
- [ ] 开发这个功能需要多久？
- [ ] 会不会延期关键节点（打板/联调/认证）？
- [ ] 同类型功能之前做过吗？还是需要从零研发？

### 风险评估
- [ ] 这个方案的成熟度如何？（用过很多次 / 用过一两次 / 全新的）
- [ ] 有供应链风险吗？（芯片独家供应？交期长？）
- [ ] 如果做不出来，有 Plan B 吗？

## 六、每个评审会的核心三问

### 方案评审时
1. 当前方案最大的技术盲区是什么？
2. 时间线上风险最大的段是哪一段（打板/联调/认证）？
3. 如果今天要砍一个功能，砍哪个？（永远准备好 Plan B）

### 原理图评审时
1. 所有外设的引脚没有冲突吗？
2. 预留了调试接口（SWD/UART）吗？
3. I2C 上拉电阻和去耦电容都画上了吗？

### 固件评审时
1. Flash/RAM 占用率是多少？留了 30% 余量吗？
2. 看门狗开启了吗？
3. 低功耗模式配置好了吗？

### 试产评审时
1. 产测方案准备好了吗？每块板子测什么？
2. 烧录方式确定了吗？（离线烧录还是产线烧录？）
3. 备件/售后方案有了吗？

## 七、最常踩的 6 个坑（及预防方法）

| 坑 | 发生在 | 预防方法 |
|----|-------|---------|
| Flash 不够 | 开发中期 | 选型时按功能清单上浮 50% Flash |
| 引脚冲突 | Layout 阶段 | 选芯片时先列外设引脚分配表 |
| 功耗超标 | 联调后期 | 立项时就明确续航目标，核算理论功耗 |
| WiFi 配网失败率高 | 用户端 | 用 BLE+WiFi 双模配网，成功率从 70%→98% |
| 天线性能差 | 结构验证 | Layout 阶段要求天线下方净空，外壳结构配合 |
| 认证不通过 | 量产前 | 预留认证预算和时间，打板阶段就考虑 EMC |

## 八、Figma + AI 快速出原型

### 15 分钟出原型的流程

1. **纸笔画线框图**（5 分钟）—— 画 3-5 个核心页面框架
2. **写 AI Prompt**（5 分钟）—— 把页面布局描述发给 AI 工具
3. **Figma 实现**（5 分钟）—— 用 Component + Auto Layout 快速搭建

### AI Prompt 模板

```
设计一个 [智能设备类型] 的 App 控制页面。

核心功能：
1. [功能 1]
2. [功能 2]
3. [功能 3]

用户需要在这个页面上完成：
- [任务 1]
- [任务 2]

设计风格：[简洁/活泼/专业]
主要色彩：[参考]
```

### 硬件 App 的核心页面套餐

硬件配套 App 通常只需要 5-15 个页面，比纯 App 少得多：

```
必做（P0）：
└── 配网引导页 —— 第一次使用流程
└── 主控制页 —— 最核心的操作页面
└── 设置页 —— 定时/通知/固件升级

选做（P1）：
└── 设备列表页 —— 如果有多个设备
└── 历史数据页 —— 如果产品有数据记录
└── 个人中心页 —— 账户/反馈/关于
```

**PM 省钱建议：** 灰度线框图（黑白、无图标）足够给开发讲清楚交互。高保真留给设计师精修。

## 附：PM 的 5 个反向提问

当工程师说"做不了"时，你可以问：

1. **"是芯片硬件不支持，还是开发时间不够？"**
   → 前者要换平台，后者加人加时间

2. **"换一个更贵的芯片能解决吗？大概加多少成本？"**
   → 很多时候问题出在成本约束，不是技术

3. **"功能上简化一点（降精度/延长时间/减功能），能做吗？"**
   → 给工程师降级选项，不是一刀切的"做不了"

4. **"同行的产品是怎么做到的？"**
   → 暗示功能可行，需要找到技术路径

5. **"这个风险的概率多大？最坏情况是什么？"**
   → 区分"有可能出问题"和"大概率会出问题"
