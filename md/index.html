﻿# 嵌入式系统知识图谱

一份从软件工程师视角出发，兼顾产品经理理解需求的嵌入式知识全景图。

第一部分：硬件基础层（MCU 与架构）

MCU 核心部件

  • 内核 (Core) — ARM Cortex-M0/M3/M4/M7 / RISC-V
    • 哈佛架构 vs 冯诺依曼架构（指令总线与数据总线是否分离）
    • 流水线 (Pipeline): 取指 → 译码 → 执行
  • 存储器
    • Flash（存放代码，掉电不丢失）
    • SRAM（运行时数据，掉电丢失）
    • 寄存器 (Register) — 外设控制的核心，本质是 SRAM 里的特殊地址
  • 时钟系统 (Clock)
    • HSI/HSE（内部/外部高速晶振）
    • PLL（锁相环，用于倍频得到更高的系统时钟）
    • 系统时钟树 (Clock Tree)
  • 电源管理 — 睡眠/停止/待机模式，功耗分级

MPU vs MCU

MCU（微控制器） MPU（微处理器）
多少钱 几块到几十块 几十到几百，还要外挂内存芯片
能跑啥系统 裸机或 FreeRTOS 这类轻量系统 完整的 Linux / Android
开发周期 短，几周可出原型 长，需要 Linux 驱动团队支持
能干多复杂的事 传感器采集、电机控制、简单显示交互 复杂网络协议栈、图形界面、视频处理
典型产品 智能手表、空调遥控器、电动车窗控制 智能音箱、扫码支付终端、车载中控大屏

选型口诀： 能用 MCU 解决的问题不要上 MPU——成本省一半，周期短一半。但如果产品需要 WiFi 配网 + 图形界面 + OTA 升级，MCU 会累死开发团队。

常见 ARM Cortex-M 系列

系列 定位 典型场景
M0/M0+ 极致省电，一颗纽扣电池跑一年 温度传感器、电子标签、基础手环
M3/M4 主力型号，M4 带浮点运算和 DSP 指令 电机控制、音频采集、绝大多数 IoT 产品
M7 高性能，带 Cache 无人机飞控、高端工业网关、复杂信号处理

PM 视角： 芯片选型不只是技术决策——它决定了供应链替代方案、缺货时的备胎选择、以及 BOM 成本天花板。

第二部分：嵌入式软件运行模型

三种编程模式（从抽象到裸核）

  1. HAL/LL 库 — 芯片厂商封装好的 API，上手快，适合原型开发
  2. 寄存器操作 — 直接读写 *(volatile uint32_t *)0x40020C14，精确控制，性能最好
  3. CMSIS — ARM 标准化的内核访问层，介于芯片厂商和用户之间

启动流程（上电后发生了什么）

复位 → Reset_Handler → SystemInit → __main → main()

具体做的事情：

  • 复制 .data 段（初始化全局变量）
  • 清零 .bss 段（未初始化全局变量）
  • 初始化堆栈指针 SP 和栈大小

中断系统

  • NVIC（嵌套向量中断控制器） — 管理所有中断的使能、挂起和优先级
  • 中断优先级 — 可配置分组（抢占优先级 / 子优先级）
  • 中断向量表 — 异常和中断的入口地址表（Reset / HardFault / MemManage / BusFault / UsageFault → 外设 IRQ）
  • SysTick（系统滴答定时器） — RTOS 的心跳

链接脚本 (Linker Script)

  • 定义 Flash / SRAM 的地址范围
  • 指定 .text / .rodata / .data / .bss 段的存放位置
  • 软件工程师经常忽略但 Debug 时必看的东西

Flash 和 RAM：容量概念

产品类型 Flash 需求 RAM 需求
简单传感器（温度上报） 16-64 KB 4-16 KB
蓝牙设备 + APP 交互 128-256 KB 32-64 KB
WiFi 配网 + 小屏幕显示 512 KB - 2 MB 128 - 512 KB
带摄像头/音频的产品 2 MB+（常需外扩 Flash） 512 KB - 几 MB

PM 视角： Flash 不够，代码塞不下，不是优化能解决的问题。在产品定义阶段给 Flash 和 RAM 留 30-50% 的余量，是在给后面的功能迭代买保险。

第三部分：外设与通信协议

基础外设一览

外设 做什么 对应软件人的类比
GPIO 控制引脚高低电平 布尔变量，可读可写
Timer 计数、PWM 输出、输入捕获 sleep() + 周期回调
ADC 模拟电压 → 数字量 analogRead()
DAC 数字量 → 模拟电压 analogWrite()
RTC 实时时钟 time() + localtime()

通信总线对比

总线 特点 需要几根线 速度
UART 异步，点对点，老派但万能 TX/RX 两根 ~115200 bps - 几 Mbps
I2C 同步，多设备共享，地址寻址 SCL/SDA 两根 ~100 kHz - 3.4 MHz
SPI 同步，全双工，高速 SCK/MOSI/MISO/CS 四根 ~几十 MHz
CAN 差分信号，工业/汽车可靠 CAN_H/CAN_L 两根 ~1 Mbps
USB 复杂但通用的 PC 连接 D+/D- 两根 可到 Gbps

PM 视角：功能和成本的对应关系

功能需求 对应接口 成本/实现影响
按键、指示灯、继电器 GPIO 几乎零成本，每个芯片都有
手机 APP 通信（蓝牙） UART + 蓝牙模块或 SoC BOM 增加约 5-20 元
测量电池电压、温度值 ADC 芯片自带，零成本
驱动小屏幕（OLED/墨水屏） SPI 或 I2C 屏幕本身贵，接口免费
控制电机转速/灯光亮度 PWM 芯片自带，零成本
多设备间通信（如分布式传感器组） CAN 总线 需 CAN 收发器，增加约 2-5 元
远程升级固件 OTA（需 WiFi 或 4G 模块） 需网络模块，Flash 至少留 1-2 MB

核心理解： 外设接口不是逐个加价的——芯片出厂时 GPIO/UART/I2C/SPI/PWM/ADC 基本都是内置的。真正的大头成本在外接的传感器模块、通信模块（WiFi/BLE/4G）、显示屏上。

逻辑分析仪

软件工程师最该买的调试工具。它能：

  • 看到 UART 的起始位 / 停止位
  • 解码 I2C 的 Start / ACK / Stop 时序
  • 比示波器便宜，更适合数字信号调试

第四部分：RTOS（实时操作系统）

核心概念

  • 任务 — 每个 while(1) 循环分时执行（不是进程，是线程的轻量版）
  • 调度器 — 通常基于优先级的抢占式调度
  • 上下文切换 — 保存/恢复寄存器 + PSP，原理和 setjmp/longjmp 很像

任务间通信

机制 类比
队列 (Queue) 类似管道 / channel
信号量 (Semaphore) 类似条件变量的计数版
互斥量 (Mutex) 有优先级继承的锁

典型问题（嵌入式面试高频）

  • 优先级反转 — 高优先级任务等待低优先级任务释放锁时，被中等优先级任务插队
  • 看门狗 (Watchdog) — 硬件定时器，到期会复位 MCU；任务必须在周期内"喂狗"
  • 临界区 (Critical Section) — 关中断保护共享数据，必须短

常见 RTOS

  • FreeRTOS — 最广泛，开源，ST/ESP 官方支持
  • RT-Thread — 国产，生态不错
  • Zephyr — Linux 基金会，现代化

PM 视角：RTOS vs 裸机

裸机 RTOS
开发速度 快 慢，但后期维护成本低
适合场景 功能固定的产品 需要同时处理多个任务的产品
典型例子 一个简单的温控器、智能灯泡 同时读传感器 + 刷屏幕 + 响应按键 + 蓝牙通信

一句话建议： 如果你的产品有超过 3 个"同时要做的事"，建议一开始就选 RTOS，否则裸机后面重构会重写一遍。

第五部分：工具链与工程

编译工具链

.c → 预处理器(cpp) → .i → 编译器 → .s → 汇编器(as) → .o → 链接器(ld) → .elf → .bin/.hex

常用文件格式

  • .elf — 带调试信息（GDB 调试用）
  • .bin — 纯二进制，从 0x08000000 烧录
  • .hex — Intel Hex 格式，带地址信息

调试方法

方法 说明

SWO 单线输出，代替 printf
Semihosting 目标端 printf 通过调试器转到 PC 控制台
串口 + Putty/TeraTerm 最传统的打日志方式

开发环境

  • STM32CubeIDE — 官方 IDE，Eclipse 改，开箱即用
  • Keil MDK / IAR — 商业 IDE，工业界老项目常用
  • VS Code + Cortex-Debug + Makefile/CMake — 自由组合

第六部分：软件工程在嵌入式中的特殊约束

内存约束

  • Flash 几 KB 到几 MB，SRAM 几 KB 到几百 KB —— 远比你的电脑小
  • 动态分配（malloc）不推荐 → 用静态池 / 定长分配器
  • Volatile 关键字 — 告诉编译器不要优化对该变量的读写，用于寄存器 / ISR 共享变量

可靠性

  • 看门狗 — 硬件级防死锁
  • CRC Checksum — Flash 完整性校验
  • 双 Bank 升级（OTA A/B Swap） — 升级失败时能回滚

低功耗

  • 不同的睡眠模式对应不同的唤醒源和唤醒延迟
  • 典型场景：传感器采集 → 睡眠 → 定时唤醒 → 采集 → 睡眠

PM 视角： 如果产品标榜"电池续航一年"，低功耗调优的时间不要砍——不是写代码，是拿万用表测电流、逐个外设关掉排查谁在偷偷耗电。

一个产品从零到一的 Timeline

W1-W2   选型 + 画板子        PM 确认功能清单，芯片采购下单
W3-W4   打板 + 焊接          等 PCB 回来，等芯片到货
W5-W8   裸机功能开发         点灯 → 串口 → 各传感器调通

W13-W14 低功耗调优           白天测电流，晚上改代码
W15-W16 小批量 + 产测        烧录脚本、出厂测试、包装
W17-W20 认证送测             蓝牙/WiFi/EMC 认证
W21-W24 试产 + 爬坡          跟踪良率，修复量产 Bug

每个节点之间没有缓冲，一个环节延误就会压到后面的排期。PM 最好的策略是把"打板等芯片"这种不可压缩的时间前置，同时给联调和低功耗留足余量。

常见的"坑"（PM 知道能帮工程师排雷）

坑 解释 对产品的影响
Flash 不够 加了新功能，固件超过芯片容量 要么砍功能，要么换更贵的芯片
低功耗不达标 理论上待机 10μA，实测 500μA 续航标一年实际三个月，口碑崩
信号干扰 WiFi 和 BLE 共用天线，同频干扰 配网失败率高，退货增加
看门狗复位 程序卡死但硬件自动重启 用户感觉"死机了"
OTA 升级死掉 升级中途断电，板子变砖 需召回或售后上门
晶振不启振 时钟没跑，MCU 不工作 板子完全不响应，制造端极难定位

第 1 课：点一个灯（HAL 库版）

目标

让 Nucleo 板上自带的 LED（通常接在 PA5 引脚）以 1Hz 闪烁。

原理

GPIO（通用输入输出）：MCU 通过寄存器控制引脚输出高电平（3.3V）或低电平（0V）。LED 接在 PA5 上，高电平亮，低电平灭。

接线

本课不需要额外接线。Nucleo 板载 LED 已经接好了：

  • LD2（绿色 LED）→ PA5 引脚
  • 串联一个限流电阻到 GND

代码（main.c）

#include "main.h"

int main(void)
{
    // 1. 初始化 HAL 库
    HAL_Init();

    // 2. 配置系统时钟为 84MHz（HSE + PLL）
    SystemClock_Config();

    // 3. 配置 GPIO
    //    - 引脚: GPIO_PIN_5 (PA5)
    //    - 模式: GPIO_MODE_OUTPUT_PP（推挽输出）
    //    - 速度: GPIO_SPEED_FREQ_LOW
    //    - 上下拉: GPIO_NOPULL（无上下拉）
    __HAL_RCC_GPIOA_CLK_ENABLE();  // 打开 GPIOA 的时钟

    GPIO_InitTypeDef gpio = {0};
    gpio.Pin   = GPIO_PIN_5;
    gpio.Mode  = GPIO_MODE_OUTPUT_PP;  // 推挽输出
    gpio.Speed = GPIO_SPEED_FREQ_LOW;
    gpio.Pull  = GPIO_NOPULL;
    HAL_GPIO_Init(GPIOA, &gpio);

    // 4. 主循环：500ms 亮，500ms 灭
    while (1)
    {
        HAL_GPIO_WritePin(GPIOA, GPIO_PIN_5, GPIO_PIN_SET);   // 亮
        HAL_Delay(500);                                        // 等 500ms

        HAL_GPIO_WritePin(GPIOA, GPIO_PIN_5, GPIO_PIN_RESET); // 灭
        HAL_Delay(500);
    }
}

代码逐行说明

  • HAL_Init() — 配置 Flash 预取、SysTick 定时器（HAL_Delay 的时钟源）
  • __HAL_RCC_GPIOA_CLK_ENABLE() — GPIO 的时钟必须先打开，否则读写 GPIO 寄存器无效。这是新手最容易忘的。
  • HAL_GPIO_Init() — 结构体配置：哪个引脚、输出还是输入、速度、是否上下拉
  • HAL_GPIO_WritePin() — 写高或写低
  • HAL_Delay(ms) — 阻塞延时，基于 SysTick 中断。精度 ~ 1ms

第 2 课：点一个灯（寄存器版）

目标

不用 HAL 库，直接操作寄存器让同一个 LED 闪烁。理解 HAL_GPIO_WritePin 背后在做什么。

原理

GPIO 通过一组寄存器控制：

寄存器 全称 做什么
MODER Mode Register 配置引脚模式（输入/输出/复用/模拟）
OTYPER Output Type Register 推挽还是开漏
OSPEEDR Output Speed Register 输出速度
PUPDR Pull-up/down Register 内部上下拉
ODR Output Data Register 写输出电平（读也是读这个值）
IDR Input Data Register 读输入电平
BSRR Bit Set/Reset Register 原子操作置位/复位（比写 ODR 安全）

代码

#include "main.h"

// GPIOA 的基地址从参考手册查得：0x40020000
// 各寄存器的偏移量（ARM 标准）：
//   MODER  @ 0x00
//   OTYPER @ 0x04
//   OSPEEDR@ 0x08
//   PUPDR  @ 0x0C
//   ODR    @ 0x14
//   BSRR   @ 0x18

#define GPIOA_MODER   (*(volatile uint32_t *)0x40020000)
#define GPIOA_ODR     (*(volatile uint32_t *)0x40020014)
#define GPIOA_BSRR    (*(volatile uint32_t *)0x40020018)
#define RCC_AHB1ENR   (*(volatile uint32_t *)0x40023830)

int main(void)
{
    // 1. 使能 GPIOA 时钟
    //    RCC_AHB1ENR 的 bit0 是 GPIOA 的时钟使能位
    RCC_AHB1ENR |= (1 << 0);

    // 2. 配置 PA5 为输出模式
    //    MODER 每 2 个 bit 配置一个引脚：
    //      00 = 输入, 01 = 输出, 10 = 复用, 11 = 模拟
    //    PA5 对应 MODER 的 bit[11:10]（因为 5*2=10）
    GPIOA_MODER &= ~(3 << 10);  // 先清零这两位
    GPIOA_MODER |=  (1 << 10);  // 设置为 01（输出模式）

    // 3. 主循环：闪烁
    while (1)
    {
        // 写 BSRR：低 16 位是置位（写 1 对应引脚输出高）
        //          高 16 位是复位（写 1 对应引脚输出低）
        GPIOA_BSRR = (1 << 5);  // PA5 输出高 → LED 亮
        for (volatile uint32_t i = 0; i < 500000; i++);  // 粗略延时

        GPIOA_BSRR = (1 << (5 + 16));  // PA5 输出低 → LED 灭
        for (volatile uint32_t i = 0; i < 500000; i++);
    }
}

代码逐行说明

  • (volatile uint32_t *)0x40020000 — 把地址 0x40020000 解释成一个 volatile 无符号 32 位整数的指针。volatile 告诉编译器不要优化对这个地址的读写——因为寄存器的值可能被硬件改变？
  • RCC_AHB1ENR |= (1 << 0) — 位操作：GPIOA 的时钟使能位是 bit0。不使能时钟，写 GPIO 寄存器无效（硬件设计的省电机制）
  • GPIOA_MODER &= ~(3 << 10) — 先清零 PA5 对应的两个 bit（10-11）
  • GPIOA_MODER |= (1 << 10) — 再置 bit10 为 1，bit11 保持 0，得到 01（输出模式）
  • for (volatile uint32_t i... — 空循环延时。volatile 防止编译器优化掉这个循环

寄存器版 vs HAL 版的对比

HAL_GPIO_WritePin(GPIOA, GPIO_PIN_5, GPIO_PIN_SET)
    ↓ 等价于
GPIOA->BSRR = GPIO_PIN_5

HAL_GPIO_Init(GPIOA, &gpio)
    ↓ 等价于
GPIOA->MODER  = xxx;
GPIOA->OTYPER = xxx;
GPIOA->OSPEEDR = xxx;
GPIOA->PUPDR  = xxx;

预期现象

和上一课一样，LED 以约 1Hz 闪烁（但空循环延时不准，实际频率会有偏差）。

常见坑

坑 解决

第 3 课：按键中断

目标

按下板载按键（PC13），LED 切换亮灭。用外部中断，而不是轮询。

原理

外部中断/事件控制器（EXTI）可以检测 GPIO 引脚上的上升沿、下降沿或双边沿，触发对应的中断服务函数（ISR）。NVIC（嵌套向量中断控制器）负责中断优先级的裁决。

Nucleo 板上的蓝色按键接在 PC13，按下时电平变低（下降沿）。

接线

不需要额外接线。板载按键：

  • 用户按键（B1）→ PC13 → 按下接地（低电平有效）

代码

#include "main.h"

// 全局变量：记录 LED 状态
volatile uint8_t led_state = 0;

int main(void)
{
    HAL_Init();
    SystemClock_Config();

    // 配置 PA5 为输出（LED）
    __HAL_RCC_GPIOA_CLK_ENABLE();
    GPIO_InitTypeDef gpio = {0};
    gpio.Pin   = GPIO_PIN_5;
    gpio.Mode  = GPIO_MODE_OUTPUT_PP;
    gpio.Speed = GPIO_SPEED_FREQ_LOW;
    gpio.Pull  = GPIO_NOPULL;
    HAL_GPIO_Init(GPIOA, &gpio);

    // 配置 PC13 为中断输入
    __HAL_RCC_GPIOC_CLK_ENABLE();
    gpio.Pin   = GPIO_PIN_13;
    gpio.Mode  = GPIO_MODE_IT_FALLING;  // 下降沿触发中断
    gpio.Pull  = GPIO_PULLUP;           // 内部上拉，不按时是高电平
    HAL_GPIO_Init(GPIOC, &gpio);

    // 配置 NVIC：使能 EXTI15_10 中断（PC13 属于这组）
    HAL_NVIC_SetPriority(EXTI15_10_IRQn, 0, 0);  // 优先级最高
    HAL_NVIC_EnableIRQ(EXTI15_10_IRQn);

    // LED 初始状态：灭
    HAL_GPIO_WritePin(GPIOA, GPIO_PIN_5, GPIO_PIN_RESET);

    while (1)
    {
        // 主循环什么都不做，中断会处理
        // 但在实际产品中，主循环会做其他事情
        HAL_Delay(1000);
    }
}

// 中断服务函数：EXTI 线 10-15 共享一个中断入口
void EXTI15_10_IRQHandler(void)
{
    // 检查是不是 PC13 触发的中断
    if (__HAL_GPIO_EXTI_GET_IT(GPIO_PIN_13) != RESET)
    {
        // 切换 LED 状态
        led_state = !led_state;
        HAL_GPIO_WritePin(GPIOA, GPIO_PIN_5,
            led_state ? GPIO_PIN_SET : GPIO_PIN_RESET);

        // 清除中断标志位（必须！否则会一直重入）
        __HAL_GPIO_EXTI_CLEAR_IT(GPIO_PIN_13);
    }
}

代码逐行说明

  • GPIO_MODE_IT_FALLING — 配置为中断模式，下降沿触发。当引脚从高电平变到低电平时，硬件产生中断请求
  • HAL_NVIC_SetPriority() — 设置中断优先级。两个参数：抢占优先级和子优先级。越小越优先
  • EXTI15_10_IRQn — 这是中断通道号。PC13 的 EXTI 线是 13，属于 10-15 这组
  • __HAL_GPIO_EXTI_CLEAR_IT() — 清中断标志。不清会反复进中断，这是典型 bug

预期现象

按一下蓝色按键，LED 亮；再按一下，LED 灭。主循环里 HAL_Delay 仍然在执行，但按键响应不会被阻塞。

常见坑

坑 现象 解决
没清中断标志 进一次中断后死机，或者按键只触发一次就一直跑中断 检查 ISR 里调用了 __HAL_GPIO_EXTI_CLEAR_IT()
按键抖动 按一次灯闪了两次 硬件加电容消抖，或软件加 20-50ms 延时消抖（延时消抖：进中断后等 20ms 再读一次状态）
优先级设错了 多个中断互相干扰 检查 HAL_NVIC_SetPriority 的参数
中断函数名拼错 中断不触发或编译 Warning EXTI15_10_IRQHandler 是固定名字，去 startup.s 文件里查向量表确认

第 4 课：PWM 呼吸灯

目标

让 LED 从暗到亮再暗，模拟呼吸效果。使用定时器的 PWM 输出功能。

原理

PWM（脉冲宽度调制）：在一个固定周期内，控制高电平的宽度（占空比）。占空比从 0% 到 100%，LED 的亮度从暗到亮。

定时器的自动重载寄存器（ARR）决定 PWM 的周期。捕获/比较寄存器（CCR）决定占空比。

接线

不需要额外接线。使用板载 LED（PA5），但要注意 PA5 是否支持 TIM2 的通道 1 输出（查 datasheet 的 AF 表确认）。

Nucleo-F103RB 的 PA5 复用为 TIM2_CH1。

代码

#include "main.h"

TIM_HandleTypeDef htim2;

void MX_TIM2_Init(void)
{
    TIM_OC_InitTypeDef sConfigOC = {0};

    // 1. 使能 TIM2 时钟
    __HAL_RCC_TIM2_CLK_ENABLE();

    // 2. 配置定时器基础参数
    htim2.Instance               = TIM2;
    htim2.Init.Prescaler         = 84 - 1;    // 84MHz / 84 = 1MHz
    htim2.Init.CounterMode       = TIM_COUNTERMODE_UP;
    htim2.Init.Period            = 1000 - 1;  // 1MHz / 1000 = 1kHz PWM
    htim2.Init.ClockDivision     = TIM_CLOCKDIVISION_DIV1;
    htim2.Init.AutoReloadPreload = TIM_AUTORELOAD_PRELOAD_ENABLE;
    HAL_TIM_PWM_Init(&htim2);

    // 3. 配置 PWM 通道
    sConfigOC.OCMode     = TIM_OCMODE_PWM1;      // PWM 模式 1
    sConfigOC.Pulse      = 0;                     // 初始占空比 0%
    sConfigOC.OCPolarity = TIM_OCPOLARITY_HIGH;   // 高电平有效
    sConfigOC.OCFastMode = TIM_OCFAST_DISABLE;
    HAL_TIM_PWM_ConfigChannel(&htim2, &sConfigOC, TIM_CHANNEL_1);
}

void set_duty(uint16_t duty)
{
    // CCR 的值 = 占空比数值（0 ~ Period）
    __HAL_TIM_SET_COMPARE(&htim2, TIM_CHANNEL_1, duty);
}

int main(void)
{
    HAL_Init();
    SystemClock_Config();

    // 配置 PA5 为 TIM2_CH1 复用功能
    __HAL_RCC_GPIOA_CLK_ENABLE();
    GPIO_InitTypeDef gpio = {0};
    gpio.Pin       = GPIO_PIN_5;
    gpio.Mode      = GPIO_MODE_AF_PP;       // 复用推挽输出
    gpio.Speed     = GPIO_SPEED_FREQ_LOW;
    gpio.Alternate = GPIO_AF1_TIM2;          // 查手册确认 AF1
    HAL_GPIO_Init(GPIOA, &gpio);

    // 初始化定时器 PWM
    MX_TIM2_Init();

    // 启动 PWM 输出
    HAL_TIM_PWM_Start(&htim2, TIM_CHANNEL_1);

    uint16_t duty = 0;
    int8_t dir = 1;  // 1 = 变亮, -1 = 变暗

    while (1)
    {
        set_duty(duty);
        HAL_Delay(5);  // 每 5ms 调整一次

        duty += dir;
        if (duty >= 999) dir = -1;   // 到达 100% 反向
        if (duty <= 0)   dir = 1;    // 到达 0% 反向
    }
}

代码逐行说明

  • Prescaler = 84 - 1 — 预分频器。84MHz / 84 = 1MHz，所以计数器每 1μs 加 1
  • Period = 1000 - 1 — 从 0 数到 999，一共 1000 个脉冲。周期 = 1000 × 1μs = 1ms，即 PWM 频率 1kHz
  • __HAL_TIM_SET_COMPARE(&htim2, TIM_CHANNEL_1, duty) — 设置 CCR 值，改变占空比。duty=0 时全灭，duty=999 时全亮
  • GPIO_AF1_TIM2 — 把 PA5 的复用功能从 GPIO 切换到 TIM2_CH1。不设这步，引脚输出不了 PWM。查 STM32 参考手册的 Alternate Function 表确认 AF 编号
  • HAL_TIM_PWM_Start() — 使能定时器的 PWM 输出，硬件开始自动产生 PWM 波形

预期现象

LED 从暗渐亮到最亮，再渐暗到灭，反复循环，像呼吸一样。

常见坑

坑 现象 解决
引脚复用功能没设 灯不亮 确认 gpio.Mode = GPIO_MODE_AF_PP 和 gpio.Alternate = GPIO_AF1_TIM2
AF 编号错了 LED 不响应 打开 datasheet 查 PA5 对应的 AF 表，有的是 AF1 有的是 AF2
Prescaler 写 0 计数器跑太快，几乎没有亮度变化 Prescaler 写得越大，PWM 周期越长。84-1 是 1μs 步进

第 5 课：UART 串口打印 "Hello"

目标

让开发板通过串口线向电脑发送 "Hello, Embedded!"。这是最简单的"调试输出"方式——嵌入式世界的 printf。

原理

UART（通用异步收发器）是一种异步串行通信协议。用两根线：TX（发送）和 RX（接收）。没有时钟线，双方约定相同的波特率（bps）来采样。

Nucleo 板上的 ST-Link 提供了一个虚拟串口，连接到电脑的 USB 上。UART2 的 TX（PA2）和 RX（PA3）默认接在 ST-Link 上。

接线

不需要额外接线。板载 ST-Link 虚拟串口已经连好了：

  • ST-Link USB 口 → 电脑 → 虚拟 COM 口
  • USART2_TX → PA2（板载已接好，不需要杜邦线）

在电脑上打开设备管理器，端口（COM 和 LPT）下能看到 "STMicroelectronics Virtual COM Port"。

代码

#include "main.h"
#include <stdio.h>  // 为了 sprintf

UART_HandleTypeDef huart2;

void MX_USART2_UART_Init(void)
{
    huart2.Instance          = USART2;
    huart2.Init.BaudRate     = 115200;            // 波特率
    huart2.Init.WordLength   = UART_WORDLENGTH_8B;
    huart2.Init.StopBits     = UART_STOPBITS_1;
    huart2.Init.Parity       = UART_PARITY_NONE;
    huart2.Init.Mode         = UART_MODE_TX_RX;
    huart2.Init.HwFlowCtl    = UART_HWCONTROL_NONE;
    huart2.Init.OverSampling = UART_OVERSAMPLING_16;
    HAL_UART_Init(&huart2);
}

// 重定向 printf 到串口
int __io_putchar(int ch)
{
    HAL_UART_Transmit(&huart2, (uint8_t *)&ch, 1, 100);
    return ch;
}

int main(void)
{
    HAL_Init();
    SystemClock_Config();

    // GPIO 和 UART 初始化
    MX_GPIO_Init();        // CubeIDE 生成的 GPIO 初始化
    MX_USART2_UART_Init();

    char buf[64];

    while (1)
    {
        sprintf(buf, "Hello, Embedded! Tick = %lu\r\n", HAL_GetTick());
        HAL_UART_Transmit(&huart2, (uint8_t *)buf, strlen(buf), 100);

        // 如果重定义了 __io_putchar，也可以用 printf
        // printf("Tick = %lu\r\n", HAL_GetTick());

        HAL_Delay(1000);
    }
}

代码逐行说明

  • HAL_UART_Init() — 初始化 UART 外设：波特率、数据位、停止位、校验位
  • HAL_UART_Transmit() — 阻塞发送。参数：UART 句柄、数据指针、长度、超时时间
  • HAL_GetTick() — 返回自 HAL_Init() 以来的毫秒数。SysTick 中断自动递增
  • __io_putchar() — 重定向 printf 的标准输出到 UART。需要在 CubeIDE 的 Project → Properties → C/C++ Build → Settings → MCU GCC Linker → Miscellaneous 里勾选 --specs=nano.specs -u _printf_float

第 6 课：I2C 读温度传感器

目标

通过 I2C 总线从 SHT30 温湿度传感器读取温度值，通过串口打印出来。

原理

I2C 是一种同步、多主多从的串行通信协议。两根线：SCL（时钟线）和 SDA（数据线）。每个从设备有唯一的 7 位地址。

SHT30 的 I2C 地址是 0x44（ADDR 引脚接 GND）或 0x45（接 VCC）。数据手册里通常写作 0x88（8 位左移 1 位的写法）。

接线

Nucleo板          SHT30 模块
 PA9 (SCL)  ────  SCL
 PA10 (SDA) ────  SDA
 3.3V       ────  VCC
 GND        ────  GND

如果模块和 Nucleo 板供电电压不同（一个 3.3V 一个 5V），需要加电平转换。SHT30 是 3.3V，Nucleo 的 I2C 引脚也是 3.3V（F103 是 5V 容忍，但建议统一用 3.3V）。

代码

#include "main.h"
#include <stdio.h>

I2C_HandleTypeDef hi2c1;

#define SHT30_ADDR    0x44 << 1   // I2C 地址（7 位左移 1 位变成 8 位）
#define SHT30_CMD_MSB 0x2C        // 单次测量命令高字节
#define SHT30_CMD_LSB 0x06        // 单次测量命令低字节（高重复性）

void MX_I2C1_Init(void)
{
    hi2c1.Instance             = I2C1;
    hi2c1.Init.Timing          = 0x00201D2B;  // 100kHz（STM32CubeMX 自动计算）
    hi2c1.Init.OwnAddress1     = 0;
    hi2c1.Init.AddressingMode  = I2C_ADDRESSINGMODE_7BIT;
    hi2c1.Init.DualAddressMode = I2C_DUALADDRESS_DISABLE;
    hi2c1.Init.GeneralCallMode = I2C_GENERALCALL_DISABLE;
    hi2c1.Init.NoStretchMode   = I2C_NOSTRETCH_DISABLE;
    HAL_I2C_Init(&hi2c1);
}

int main(void)
{
    HAL_Init();
    SystemClock_Config();
    MX_GPIO_Init();
    MX_I2C1_Init();
    MX_USART2_UART_Init();  // 复用上一课的串口打印

    uint8_t cmd[2] = {SHT30_CMD_MSB, SHT30_CMD_LSB};
    uint8_t buf[6];  // 返回 6 字节：温度 MSB/LSB/CRC + 湿度 MSB/LSB/CRC
    float temp, hum;

    while (1)
    {
        // 1. 发送测量命令
        HAL_I2C_Master_Transmit(&hi2c1, SHT30_ADDR, cmd, 2, 100);

        HAL_Delay(20);  // 等传感器完成测量（最长 15ms）

        // 2. 读取 6 字节数据
        HAL_I2C_Master_Receive(&hi2c1, SHT30_ADDR, buf, 6, 100);

        // 3. 将原始值转换为温度和湿度
        //    温度 = -45 + 175 * (raw / 65535)
        uint16_t raw_temp = (buf[0] << 8) | buf[1];
        temp = -45.0f + 175.0f * (float)raw_temp / 65535.0f;

        uint16_t raw_hum = (buf[3] << 8) | buf[4];
        hum = 100.0f * (float)raw_hum / 65535.0f;

        // 4. 通过串口打印
        char msg[64];
        sprintf(msg, "Temp: %.2f C, Hum: %.2f %%\r\n", temp, hum);
        HAL_UART_Transmit(&huart2, (uint8_t *)msg, strlen(msg), 100);

        HAL_Delay(2000);
    }
}

代码逐行说明

  • SHT30_ADDR 0x44 << 1 — STM32 的 HAL 库要求 I2C 地址是 8 位格式（7 位地址左移 1 位，最低位是 R/W 位，由 HAL 自动填充）
  • HAL_I2C_Master_Transmit() — 主机发送：起始条件 → 设备地址（写）→ 数据 → 停止条件
  • HAL_I2C_Master_Receive() — 主机接收：起始条件 → 设备地址（读）→ 读取数据 → 停止条件
  • buf[0] << 8 | buf[1] — 把两个 8 位合并成一个 16 位。温度值的大端格式：MSB 在前
  • -45 + 175 * raw / 65535 — SHT30 数据手册上给的计算公式

预期现象

串口每秒输出一次：

Temp: 25.43 C, Hum: 58.21 %
Temp: 25.47 C, Hum: 58.15 %
...

用手指捏住传感器，温度读数会上升。

常见坑

坑 现象 解决
I2C 地址错了 HAL_I2C_Master_Transmit 返回 HAL_ERROR 检查传感器型号，查 datasheet 确认 7 位地址
没接上拉电阻 I2C 通信不稳定或失败 I2C 总线需要 4.7kΩ 上拉电阻到 3.3V。Nucleo 板的 I2C 引脚有板载上拉，但面包板接长线建议外加
Timing 参数不对 I2C 通信超时 Timing 参数由 CubeMX 根据时钟频率和 I2C 速率自动算出，不要手动改
传感器供电不足 读数异常 从 3.3V 供电，不要用 5V

第 7 课：SPI 驱动 OLED 屏幕

目标

用 SPI 总线驱动 SSD1306 OLED 屏幕，显示 "Hello" 文字。

原理

SPI 是全双工同步串行协议。四根线：

  • SCK（时钟线）
  • MOSI（主机输出，从机输入）
  • MISO（主机输入，从机输出）— OLED 通常不接，浮空
  • CS（片选，低电平有效）

SSD1306 由 128×64 个像素组成，每个像素 1 bit（亮/灭）。显示内容写入 1KB 的 GDDRAM 中。

接线

Nucleo F103RB         SSD1306 (SPI 版)
 PA5 (SCK)      ────  SCL/SCK
 PA7 (MOSI)     ────  SDA/MOSI
 PA4 (CS)       ────  CS
 PA3 (DC)       ────  DC（数据/命令选择）
 PA2 (RST)      ────  RES/RST
 3.3V           ────  VCC
 GND            ────  GND

代码

#include "main.h"
#include <stdio.h>
#include <string.h>

SPI_HandleTypeDef hspi1;

// OLED 引脚定义（用 GPIO 控制）
#define OLED_CS_PIN     GPIO_PIN_4
#define OLED_CS_PORT    GPIOA
#define OLED_DC_PIN     GPIO_PIN_3
#define OLED_DC_PORT    GPIOA
#define OLED_RST_PIN    GPIO_PIN_2
#define OLED_RST_PORT   GPIOA

// OLED 命令宏
#define OLED_CMD  0
#define OLED_DATA 1

void OLED_Write(uint8_t type, uint8_t data)
{
    HAL_GPIO_WritePin(OLED_DC_PORT, OLED_DC_PIN, type);  // DC = 0:命令, 1:数据
    HAL_GPIO_WritePin(OLED_CS_PORT, OLED_CS_PIN, GPIO_PIN_RESET);  // CS 拉低
    HAL_SPI_Transmit(&hspi1, &data, 1, 100);
    HAL_GPIO_WritePin(OLED_CS_PORT, OLED_CS_PIN, GPIO_PIN_SET);    // CS 拉高
}

void OLED_Init(void)
{
    HAL_Delay(100);

    // 硬件复位
    HAL_GPIO_WritePin(OLED_RST_PORT, OLED_RST_PIN, GPIO_PIN_RESET);
    HAL_Delay(10);
    HAL_GPIO_WritePin(OLED_RST_PORT, OLED_RST_PIN, GPIO_PIN_SET);
    HAL_Delay(10);

    // 初始化命令序列（来自 SSD1306 数据手册）
    OLED_Write(OLED_CMD, 0xAE);  // 关闭显示
    OLED_Write(OLED_CMD, 0xD5);  // 设置显示时钟分频
    OLED_Write(OLED_CMD, 0x80);
    OLED_Write(OLED_CMD, 0xA8);  // 设置多路复用比
    OLED_Write(OLED_CMD, 0x3F);  // 64
    OLED_Write(OLED_CMD, 0xD3);  // 设置显示偏移
    OLED_Write(OLED_CMD, 0x00);
    OLED_Write(OLED_CMD, 0x40);  // 设置显示开始行
    OLED_Write(OLED_CMD, 0x8D);  // 电荷泵
    OLED_Write(OLED_CMD, 0x14);  // 启用
    OLED_Write(OLED_CMD, 0x20);  // 内存寻址模式
    OLED_Write(OLED_CMD, 0x00);  // 水平寻址
    OLED_Write(OLED_CMD, 0xA1);  // 段重映射（列 127 = SEG0）
    OLED_Write(OLED_CMD, 0xC8);  // COM 扫描方向
    OLED_Write(OLED_CMD, 0xDA);  // COM 引脚配置
    OLED_Write(OLED_CMD, 0x12);
    OLED_Write(OLED_CMD, 0x81);  // 对比度
    OLED_Write(OLED_CMD, 0xCF);
    OLED_Write(OLED_CMD, 0xD9);  // 预充电周期
    OLED_Write(OLED_CMD, 0xF1);
    OLED_Write(OLED_CMD, 0xDB);  // VCOMH 失配电压
    OLED_Write(OLED_CMD, 0x40);
    OLED_Write(OLED_CMD, 0xA4);  // 全局显示开启（从 RAM）
    OLED_Write(OLED_CMD, 0xA6);  // 正常显示（非反色）
    OLED_Write(OLED_CMD, 0xAF);  // 打开显示
}

void OLED_Clear(void)
{
    // 清空 GDDRAM（128 × 64 / 8 = 1024 字节）
    for (uint16_t i = 0; i < 1024; i++)
        OLED_Write(OLED_DATA, 0x00);
}

// 6×8 字体点阵（只放一个 'H' 作为示例）
const uint8_t font_6x8_H[] = {
    0x00, 0x00, 0x7C, 0x12, 0x12, 0x7C  // H
};

void OLED_ShowChar(uint8_t x, uint8_t y, char ch)
{
    // 简单的点阵字符显示
    // 每个字符 6 列 × 8 行，对应 6 字节
    uint8_t c = ch - 32;  // ASCII 偏移
    // 实际需要完整的字库表（ASCII 码表），这里简化
    // 完整实现可以用 U8g2 库（见附录）
    OLED_Write(OLED_DATA, 0x7C); OLED_Write(OLED_DATA, 0x12);
    OLED_Write(OLED_DATA, 0x12); OLED_Write(OLED_DATA, 0x7C);
    OLED_Write(OLED_DATA, 0x12); OLED_Write(OLED_DATA, 0x12);
}

void OLED_ShowString(uint8_t x, uint8_t y, const char *str)
{
    while (*str) {
        OLED_ShowChar(x, y, *str);
        x += 6;
        if (x > 122) { x = 0; y += 8; }
        str++;
    }
}

int main(void)
{
    HAL_Init();
    SystemClock_Config();
    MX_GPIO_Init();
    MX_SPI1_Init();

    OLED_Init();
    OLED_Clear();

    // 在第 0 行第 16 列显示 "Hello"
    // 注意：需要完整的 6x8 字库才能正常显示

    for (uint8_t i = 0; i < 5; i++)

    while (1)
    {
        HAL_Delay(1000);
    }
}

代码逐行说明

  • OLED_Write() — SPI 写一字节，先控制 DC 引脚区分命令还是数据，CS 拉低选中设备，发送完成后 CS 拉高
  • 初始化命令序列 — 每个 OLED 屏需要一套特定的初始化命令，数据手册会给出。ST 的 HAL 库没有对 SSD1306 的封装，所以需要自己按手册把命令写出来
  • OLED_Clear() — 往 GDDRAM 写 1024 个 0x00，清屏
  • OLED_ShowChar() — 在指定位置显示一个字符。这里用了简化的实现，实际项目中建议用 u8g2 库（见附录）

第 8 课：ADC 采样

目标

读取电位器的模拟电压值，通过串口打印。

原理

ADC（模数转换器）把连续的模拟电压转换成离散的数字量。STM32F103 的 ADC 是 12 位的，所以返回值范围 0-4095，对应电压 0V-3.3V。

接线

Nucleo F103RB     电位器（10kΩ）
 PA0 (ADC1_IN0) ──── 中间抽头（wiper）
 3.3V           ──── 一端
 GND            ──── 另一端

代码

#include "main.h"
#include <stdio.h>

ADC_HandleTypeDef hadc1;

void MX_ADC1_Init(void)
{
    hadc1.Instance                   = ADC1;
    hadc1.Init.ClockPrescaler        = ADC_CLOCK_SYNC_PCLK_DIV4;
    hadc1.Init.Resolution            = ADC_RESOLUTION_12B;  // 12 位
    hadc1.Init.ScanConvMode          = DISABLE;
    hadc1.Init.ContinuousConvMode    = ENABLE;   // 连续转换
    hadc1.Init.DiscontinuousConvMode = DISABLE;
    hadc1.Init.ExternalTrigConv      = ADC_SOFTWARE_START;
    hadc1.Init.DataAlign             = ADC_DATAALIGN_RIGHT;
    hadc1.Init.NbrOfConversion       = 1;
    hadc1.Init.DMAContinuousRequests = DISABLE;
    hadc1.Init.EOCSelection          = ADC_EOC_SINGLE_CONV;
    HAL_ADC_Init(&hadc1);
}

int main(void)
{
    HAL_Init();
    SystemClock_Config();
    MX_GPIO_Init();
    MX_ADC1_Init();
    MX_USART2_UART_Init();

    // 启动 ADC 校准（提高精度）
    HAL_ADCEx_Calibration_Start(&hadc1);

    // 启动连续转换
    HAL_ADC_Start(&hadc1);

    uint32_t adc_value;
    float voltage;
    char msg[64];

    while (1)
    {
        // 等待转换完成
        if (HAL_ADC_PollForConversion(&hadc1, 100) == HAL_OK)
        {
            adc_value = HAL_ADC_GetValue(&hadc1);
            voltage   = adc_value * 3.3f / 4095.0f;

            sprintf(msg, "ADC: %lu, Voltage: %.2f V\r\n",
                    (unsigned long)adc_value, voltage);
            HAL_UART_Transmit(&huart2, (uint8_t *)msg, strlen(msg), 100);
        }

        HAL_Delay(500);
    }
}

代码逐行说明

  • ADC_RESOLUTION_12B — 12 位精度，返回值 0-4095
  • ContinuousConvMode = ENABLE — 连续转换模式：ADC 自动不停采样，不需要每次都手动触发
  • HAL_ADCEx_Calibration_Start() — 校准 ADC（补偿内部偏移）。不做也行，但做了精度更好
  • HAL_ADC_PollForConversion() — 轮询等待转换完成。因为开了连续模式，它几乎总是立即返回
  • HAL_ADC_GetValue() — 读取转换结果寄存器
  • adc_value * 3.3f / 4095.0f — 将 ADC 原始值换算成电压。12 位 ADC 把 3.3V 分为 4096 份

预期现象

旋转电位器，串口输出的电压值在 0V 到 3.3V 之间变化。

常见坑

坑 现象 解决
ADC 初始化多次调用 卡死在 HAL_ADC_Init 只在程序开始的时候初始化一次
电位器接错了 读数一直满量程或一直 0 确认中间抽头接 PA0，两端接 VCC 和 GND
电压不准 读数偏高或偏低 校准函数要在 ADC 启动前调用，且 VREF 引脚必须接稳定的 3.3V

第 9 课：FreeRTOS 双任务

目标

在 STM32 上移植 FreeRTOS，创建两个任务：一个让 LED 以 1Hz 闪烁，一个通过串口每秒打印 "Hello from Task2"。

原理

FreeRTOS 是一个抢占式实时操作系统内核。任务基于优先级调度：高优先级的任务就绪时，会打断正在运行的低优先级任务。

上下文切换是由 SysTick 定时器（通常 1ms 或 10ms 一个 tick）触发的 PendSV 中断完成的。

接线

不需要额外接线。使用板载 LED（PA5）和 UART2（串口打印）。

代码

#include "FreeRTOS.h"
#include "task.h"
#include "main.h"

// 任务栈大小和优先级
#define TASK_LED_STACK_SIZE  128
#define TASK_LED_PRIORITY    1
#define TASK_PRINT_STACK_SIZE 256
#define TASK_PRINT_PRIORITY   1

// 任务句柄
TaskHandle_t led_task_handle = NULL;
TaskHandle_t print_task_handle = NULL;

void LED_Task(void *argument)
{
    while (1)
    {
        HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);  // 翻转 LED
        vTaskDelay(pdMS_TO_TICKS(500));          // 延时 500ms
    }
}

void Print_Task(void *argument)
{
    const char *msg = "Hello from Task2\r\n";
    while (1)
    {
        HAL_UART_Transmit(&huart2, (uint8_t *)msg, strlen(msg), 100);
        vTaskDelay(pdMS_TO_TICKS(1000));
    }
}

int main(void)
{
    HAL_Init();
    SystemClock_Config();
    MX_GPIO_Init();
    MX_USART2_UART_Init();

    // 创建两个任务
    xTaskCreate(LED_Task, "LED", TASK_LED_STACK_SIZE, NULL,
                TASK_LED_PRIORITY, &led_task_handle);
    xTaskCreate(Print_Task, "Print", TASK_PRINT_STACK_SIZE, NULL,
                TASK_PRINT_PRIORITY, &print_task_handle);

    // 启动 FreeRTOS 调度器
    vTaskStartScheduler();

    // 如果调度器启动失败，会走到这里
    while (1);
}

代码逐行说明

  • xTaskCreate() — 创建一个任务。参数：函数指针、名字、栈大小（单位 word，不是 byte）、参数、优先级、句柄
  • vTaskDelay() — 任务延时。当前任务进入阻塞状态，释放 CPU 给其他任务。参数是 tick 数，用 pdMS_TO_TICKS(ms) 将毫秒转成 tick
  • vTaskStartScheduler() — 启动调度器。调度器启动后，FreeRTOS 接管 SysTick 和 PendSV 中断，按优先级分配 CPU 时间
  • 两个任务优先级相同（都是 1）→ 时间片轮转调度。每个任务运行一个时间片（通常是 1 tick）后切换到另一个
  • pdMS_TO_TICKS() — 将毫秒转换成 tick 数。默认 tick 频率是 1000Hz（1ms/tick），所以 pdMS_TO_TICKS(500) = 500 ticks

在 CubeIDE 中启用 FreeRTOS

  1. Pinout & Configuration → Middleware → FREERTOS → Mode 选 CMSIS_V2
  2. Configuration → Tasks → 添加两个任务：LedTask 和 PrintTask
  3. 自动生成的代码会包含 FreeRTOS 的头文件和初始化
  4. 代码写在 void StartLedTask(void *argument) 和 void StartPrintTask(void *argument) 中

预期现象

  • LED 以 1Hz 闪烁（500ms 亮，500ms 灭）
  • 串口每秒打印一次 "Hello from Task2"
  • 两个任务互相不阻塞

常见坑

坑 现象 解决
vTaskDelay 传了 0 任务不释放 CPU 延时至少 1 tick。传 0 等同于 yield（让出当前时间片但不阻塞）
栈大小不够 任务运行后 HardFault FreeRTOS 默认配置下，任务栈至少 128 word（512 bytes）。Print 任务因为有 printf 要更多栈，用 256
忘记配置 SysTick FreeRTOS 不运行（一直卡在 vTaskStartScheduler） CubeMX 的 SYS → Timebase Source 不能是 SysTick（因为被 FreeRTOS 占用了），改成 TIM6 或 TIM7
优先级反转 高优先级任务响应延迟 使用互斥量（xSemaphoreCreateMutex）而不是二元信号量。互斥量有优先级继承机制

第 10 课：看门狗——让死机自愈

目标

启用独立看门狗（IWDG），让 LED 按一定频率闪烁。故意卡死程序，观察 MCU 自动复位。

原理

独立看门狗（IWDG）是一个由独立 RC 振荡器驱动的硬件定时器。它不依赖主时钟，即使主时钟挂了，IWDG 仍在计数。

一旦启动，IWDG 从预设值开始递减计数。计数到 0 时，产生系统复位。程序必须在超时前"喂狗"（重装载计数器），否则 MCU 自动重启。

接线

不需要额外接线。

代码

#include "main.h"

IWDG_HandleTypeDef hiwdg;

void MX_IWDG_Init(void)
{
    hiwdg.Instance       = IWDG;
    hiwdg.Init.Prescaler = IWDG_PRESCALER_64;   // 40kHz / 64 = 625Hz
    hiwdg.Init.Reload    = 1250;                 // 1250 / 625 = 2 秒超时
    HAL_IWDG_Init(&hiwdg);
}

int main(void)
{
    HAL_Init();
    SystemClock_Config();
    MX_GPIO_Init();
    MX_IWDG_Init();  // 启动看门狗，2 秒超时

    uint32_t count = 0;

    while (1)
    {
        HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);  // 翻转 LED
        count++;

        if (count < 5)
        {
            // 前 5 次正常喂狗
            HAL_IWDG_Refresh(&hiwdg);  // 喂狗：重置计数器
            HAL_Delay(1000);           // 延时 1 秒，在超时范围内
        }
        else
        {
            // 故意不喂狗，等 3 秒 → IWDG 超时 → MCU 复位
            // 此时 LED 会保持当前状态不动
            HAL_Delay(3000);  // 延时 3 秒，超过 2 秒的看门狗超时
            // 代码不会执行到这里，因为 MCU 已经复位了
        }
    }
}

代码逐行说明

  • IWDG_PRESCALER_64 — 预分频器 64。IWDG 的时钟源是 LSI（低速内部振荡器），约 40kHz。分频后 = 40kHz / 64 ≈ 625Hz，每个 tick 约 1.6ms
  • Reload = 1250 — 重装载值。超时时间 = 1250 / 625 = 2 秒。两秒内不喂狗就复位
  • HAL_IWDG_Refresh() — 喂狗：把重装载值写入计数器，重新开始倒计时
  • 程序运行逻辑：前 5 次循环，每 1 秒喂一次狗（在 2 秒超时内），LED 正常闪烁。第 6 次循环，不喂狗且延时 3 秒 → IWDG 超时 → MCU 复位 → 程序从头开始

预期现象

  1. LED 以 1Hz 闪烁 5 次
  2. 然后 LED 停在当前状态约 2 秒
  3. MCU 复位，LED 重新开始闪烁（从第 1 次开始）
  4. 串口不会输出任何内容（复位后重新运行 main 开头，但本课没有配串口输出）

常见坑

坑 现象 解决
喂狗太晚 看门狗总复位，设备不停重启 确保喂狗间隔小于 IWDG 的超时时间——留 50% 余量
忘记喂狗 设备间歇性重启 在任务主循环里喂狗，不要在中断里喂狗（中断里喂狗 → 主循环卡死了但中断还在跑 → 看门狗不触发）
Prescaler 计算错了 超时时间不对 IWDG 的时钟是 LSI（~40kHz），不是 HSI/HSE。不同芯片的 LSI 频率可能有 ±10% 偏差

如果你是个产品经理，第一次接触嵌入式硬件产品，你最需要的不是学会写代码或画电路图，而是建立一套心智模型：知道一个硬件产品由哪些部分拼起来的、每个部分的成本与风险在哪里、以及当工程师跟你说"这个做不了"时，你该怎么判断是真做不了还是条件没谈拢。

第一章：嵌入式产品到底是什么

1.1 从一个智能插座说起

假设我们要做一个智能插座——能用手机 App 远程开关、能显示当前功率、能设置定时。这个产品拆开来看，需要这些：

         │  │ MCU  │  │ WiFi模块  │  │电源 │  │
         │  │ (大脑) │  │ (通信)   │  │模块 │  │

         │     │        │ (执行器) │         │

一个嵌入式产品，不管多复杂，都可以拆成这四个层次来看：

层次 是什么 智能插座里的例子 成本占比（大约）
计算核心 MCU（微控制器）或 MPU（微处理器） 一个 10-15 元的 MCU 芯片 10-20%
通信能力 联网模块 WiFi 模组（8-15 元） 10-15%
传感器/执行器 感知物理世界 or 控制物理世界 电流检测芯片、继电器 15-25%

PM 要记住的核心概念： 一个嵌入式产品 = 大脑（MCU） + 通信 + 感知/控制 + 供电与外壳。这四个部分互相依赖——选哪个大脑决定了你能用哪个通信模组，电源决定了传感器能跑多快。

1.2 MCU（微控制器）是什么，不是什么

产品经理最常见的误解是：把 MCU 当成一个"小电脑"。

误解 真相
MCU 和电脑 CPU 差不多，只是弱一些 完全不同。MCU 是单芯片集成——CPU、内存、存储、外设接口全部印在同一块硅上。电脑的 CPU 只是一个核，还要配独立的内存条和硬盘
MCU 性能越强越好 错。功耗、成本、封装大小、开发复杂度都会随性能剧增。选 MCU 是在四个维度上找平衡
软件可以随便往 MCU 上加 错。MCU 的 Flash 容量是硬上限。代码超过容量 = 物理上装不进去，不是"优化一下就能省出来"

MCU 的几个硬指标，产品经理需要看懂：

参数 它实际上决定了什么
Flash 容量 你能写多少功能的代码。好比硬盘容量，单位 KB 或 MB
RAM 大小 程序同时能处理多少数据。好比内存条，单位 KB 或 MB
主频 (MHz) 芯片算得快不快。但嵌入式设备往往不是卡在算力，而是卡在 I/O
引脚数 你能接多少个外设。引脚不够 = 换更大的封装（成本涨）

技术细节保留： Flash 常见 16KB ~ 2MB，RAM 常见 4KB ~ 512KB。这是芯片定价的核心分档依据。一个 Cortex-M3 内核的 STM32F103，64KB Flash + 20KB RAM，裸片价格大约 8-15 元。

1.3 硬件和软件在嵌入式里的特殊关系

在纯软件产品里，加功能 = 加代码。在嵌入式产品里，加功能可能意味着：

  • 需要的 Flash 变大 → 换更大容量的芯片型号 → BOM 成本上升
  • 需要的引脚变多 → 换更多引脚的封装 → PCB 面积变大 → 外壳可能得重新设计
  • 需要的外设新类型（比如加蓝牙）→ 可能需要换芯片平台 → 软件开发推翻重来

这是做嵌入式产品最核心的思维转变：硬件是软件的天花板，一旦定下来改了就很贵。

第二章：芯片选型 —— 产品的天花板在哪里

2.1 第一道选择题：MCU 还是 MPU

这是最顶层的技术架构决策，决定了产品的成本基线、团队配置和开发周期。

维度 MCU（微控制器） MPU（微处理器）
芯片价格 3 - 50 元 20 - 200+ 元（还要外挂 Flash/RAM）
典型 BOM 成本（整板） 15 - 60 元 80 - 500 元
操作系统 裸机或 FreeRTOS Linux / Android
开发团队 2-4 名嵌入式工程师 需要驱动 + 系统 + 应用层工程师
原型周期 4-8 周 8-20 周
能做多复杂的事 传感器、控制、简单屏幕交互 网络协议栈、图形 UI、视频处理

PM 的核心判断逻辑：

  • ✅ 选 MCU，如果： 产品主要做采集-判断-控制循环，不需要复杂图形界面，没有需要跑第三方软件的诉求
  • ✅ 选 MPU，如果： 需要浏览器、需要复杂的 GUI 框架（Qt/Flutter）、需要运行加密协议栈或 AI 推理框架

经验法则： 如果产品的交互只是几个按键 + 一个小屏幕（≤ 3.5 寸），MCU 就够了。如果涉及触摸屏滑动、网页渲染、视频播放，上 MPU。

2.2 MCU 内核：Cortex-M 的档位（技术细节，但 PM 需要知道）

ARM Cortex-M 是 MCU 里最主流的 CPU 内核架构，几个关键档位：

系列 算力 功耗 价格档位 典型应用
M0/M0+ 低 极低（μA 级待机） 3-8 元 温度传感器、电子标签、门锁
M3 中 低 8-20 元 工控、家电、物联网中端产品
M4 中高（带浮点运算） 中 15-40 元 音频处理、电机控制、需要运算的产品

对 PM 来说： 绝大多数 IoT 产品落在 M3 和 M4 区间。M0 适合电池续航极关键的产品。M7 通常意味着产品已经很复杂了，接近"要不要上 MPU"的边界。

这是 PM 最需要看懂的硬件参数表。

现实情况： 开发到中期发现 Flash 不够用，是嵌入式产品最常见的延期原因之一。不是工程师没写好代码，而是功能需求超过了芯片的物理容量。这时候的选择只有两个——砍功能 或 换更贵的芯片。

第三章：产品功能如何落地到硬件

3.1 理解"接口"和"引脚"

MCU 是一片芯片，它通过引脚和外面的世界连接。每个引脚可以做特定的事。一块有 48 个引脚的芯片和一块 100 个引脚的芯片，价格可能差两倍。

但好消息是：大多数基础功能（GPIO/UART/I2C/SPI/PWM/ADC）芯片出厂就内置了，不需要额外花钱。 真正加钱的是外接的模块——传感器、通信模组、显示屏。

3.2 一个功能对应什么硬件，要花多少钱

这是产品经理最需要记住的一张表：

你想要的功能 需要什么接口 接口是否芯片自带 额外 BOM 成本
按键 GPIO ✅ 几乎所有芯片都有 ~0 元（仅 PCB 走线）
LED 指示灯 GPIO ✅ 同上 几毛钱（LED 灯珠+限流电阻）
控制继电器（开关大电流） GPIO + 驱动电路 ✅ GPIO 自带，驱动电路需外加 2-5 元
通过手机 App 控制（蓝牙） UART ✅ UART 自带 5-20 元（蓝牙模块）
设备连 WiFi SPI 或 SDIO ✅ 部分芯片自带 WiFi 8-25 元（WiFi 模组）
测量电池电压 ADC ✅ 大多数芯片有 ~0 元（分压电阻）
测温度/湿度 I2C ✅ I2C 自带 2-10 元（传感器芯片）
驱动 OLED 屏幕 SPI 或 I2C ✅ 自带 屏幕 5-30 元
让电机转起来或调亮度 PWM ✅ 大多数芯片有 驱动芯片 + MOS 管 3-10 元
测距离（超声波/红外） GPIO + Timer ✅ 自带 传感器模块 5-15 元
播放简单提示音 PWM + DAC ✅ 大多数有 喇叭 1-5 元
设备间长距离通信 CAN 部分芯片自带 CAN 控制器 CAN 收发器 2-5 元

技术细节： 每条通信协议都有物理层差异。UART 需要两根线（TX/RX），I2C 需要两根线（SCL/SDA）且支持多设备挂在一根总线上，SPI 需要四根线（SCK/MOSI/MISO/CS）且速度最快。这些细节在跟硬件工程师确认引脚分配时会用到。

3.3 现实产品案例：拆一个智能门锁的 BOM

有了上面的概念，我们可以看一个真实产品的硬件成本构成（仅供参考，量级对）：

部件 接口 估算成本
MCU（带低功耗蓝牙） — 12 元
指纹传感器 SPI 15 元
蓝牙天线 + 匹配电路 射频引脚 2 元
电机驱动（开锁） PWM + GPIO 5 元
电池管理 + 充电 ADC 检测电压 3 元

这个例子想说的是：MCU 本身（12 元）只占整机 BOM 的 15-20%，但 MCU 选错了，其他所有外设都得跟着变。这就是为什么芯片选型是产品定义阶段最重要的决策之一。

3.4 硬件的物理限制：大小和功耗

产品经理的 PRD 里经常写"体积尽量小、续航尽量长"。但这两个指标在实际中是互相矛盾、且都有明确物理下限的：

关于体积：

  • MCU 封装从 3mm×3mm 到 14mm×14mm
  • 天线需要特定面积（蓝牙天线至少 15mm×6mm）
  • 电池体积直接决定容量
  • 多层 PCB 比双层贵 2-3 倍，但能做更小的面积

关于功耗（技术细节但很有用）：

  • MCU 工作电流：mA 级
  • MCU 待机电流：μA 级（几微安）
  • WiFi 发射峰值：100-300 mA（这个很耗电）
  • 蓝牙 BLE 工作电流：5-15 mA
  • 传感器工作电流：几十 μA 到几 mA

快速估算： 一块 500mAh 的电池，如果设备工作平均电流 10mA，理论续航 = 500 ÷ 10 = 50 小时 ≈ 2 天。如果要续航一年，平均电流必须小于 500 ÷ (365×24) ≈ 0.057mA = 57μA。这就是为什么长续航设备必须大部分时间处于睡眠模式。

第四章：软件架构决策 —— 裸机、RTOS、还是 Linux

4.1 为什么 PM 也要理解软件架构

软件架构的选择直接影响开发周期、功能上限和后期维护成本。它是 PM 和工程团队之间最容易被忽略的关键对齐节点。

4.2 三种方案的对比

裸机 (Bare Metal) RTOS（实时操作系统） Linux（完整 OS）
本质 一个 while 循环跑所有逻辑 多个任务轮换执行 多进程多线程
开发难度 低 中 高
功能上限 低（任务多了代码极其难维护） 中高 高
内存占用 几乎无额外开销 RTOS 内核 ~ 几 KB 至少 8-16 MB
复杂度临界点 当你要同时处理 >3 件事时 当实时性要求极高或后续迭代频繁时 当需要复杂网络/图形/多媒体时
调试难度 容易 中等 复杂（需要 Linux 驱动经验）
量产风险 低 低 中（系统复杂度带来偶发 bug）
典型代表 遥控器、电子秤、温控器 智能家居网关、医疗设备 智能音箱、车载中控

技术细节： RTOS 不会让 MCU 变快，但它能让多个"同时要做的事情"以确定性的方式轮流执行。如果裸机单循环里一个传感器读取阻塞了 100ms，整个系统的按键响应、屏幕刷新都会一起卡。RTOS 把每个功能拆成独立任务，一个卡住不影响其他的。

4.3 决策树：你的产品该选哪种

产品需要做的功能有没有超过 3 个"同时发生的事"？
├── 没有 → 裸机就够了（开发最快，成本最低）
└── 有 → 是否需要复杂图形界面 / 网页 / 第三方软件？
    ├── 是 → 上 Linux（但需要 MPU，BOM 成本大幅上升）
    └── 否 → 上 RTOS（MCU 照用，开发周期增加 2-4 周但值得）

4.4 RTOS 里的几个关键概念（PM 需知）

概念 一句话解释 为什么 PM 要知道
任务/线程 每个 while(1) 做的事（读传感器、刷屏幕、通信） 产品功能越多，任务越多，RTOS 的优势越明显
抢占式调度 高优先级的事能打断低优先级的事 但可能出现优先级反转——高优先级等低优先级时被中优先级插队，导致关键功能延迟
看门狗 硬件定时器，程序卡死了就自动重启 这是嵌入式设备比普通电脑更可靠的原因之一——死锁了能自己活过来
队列/信号量 任务之间传递数据的方式 任务之间的"对话"越复杂，软件调试越花时间
临界区 不能被打断的代码段，通常很短 如果工程师说要"关中断保护"，意思是有一小段代码不能被其他任务打断——正常现象，但关中断太久会影响实时性

经验之谈： 很多团队在原型阶段图快用了裸机，做到后期功能爆炸才被迫重构成 RTOS——等于整个软件重写一遍。如果你的产品在需求文档阶段就已经规划了"App 配网 + OTA 升级 + 云平台交互 + 本地传感器 + 屏幕显示"五个功能模块，一开始就从 RTOS 起步。

4.5 嵌入式里的可靠性机制

机制 它做了什么 如果没做好会怎样
看门狗 硬件周期性检查程序是否还活着，死锁了自动复位 设备死机后不会自愈，用户只能拔电源
OTA 升级 通过无线网络更新固件 无法远程修 bug，产品上市后发现问题只能召回
双 Bank 升级 Flash 分成两个区，升级失败能回退到旧版本 升级中途断电 = 变砖，只能返厂
低功耗管理模式 不需要时进入睡眠模式节省电量 续航只有预期的 1/10

找外包公司 招聘自建团队
适合阶段 原型验证、小批量 长期产品线、核心产品

第六章：跟嵌入式团队高效沟通的实用手册

6.1 工程师说的这些话到底是什么意思

"Flash 打满了"

  • 字面意思：芯片的存储容量用完了，再加任何功能都装不进去
  • PM 能做的：拿出功能清单，和工程师一起砍优先级。或者同意换更大 Flash 的芯片（通常贵 2-5 元，可能会影响 PCB 兼容性）

"这个引脚不够用了"

  • 字面意思：我们需要控制的外设数量超过了芯片的物理引脚数
  • PM 能做的：接受换更多引脚的芯片（成本增加、PCB 面积变大），或者去掉一些非必要的外设

"低功耗还没调"

  • 字面意思：芯片现在处在"全速跑"模式，没有进入睡眠。这个阶段不是写代码，是逐个关掉外设测电流
  • PM 能做的：别催进度，这个是纯物理调试，基本看板和工具决定速度

"要换大封装的型号"

  • 字面意思：同样内核的芯片，引脚更多/Flash 更大的版本。PCB 可能要做改动
  • 成本影响：芯片贵 2-10 元，PCB 可能得重新 Layout，增加 1-2 周

"这个需要上 Linux"

  • 字面意思：MCU 处理不了这么复杂的功能，得换 MPU 方案
  • 成本影响：BOM 可能翻倍，开发周期延长 1-2 个月，团队需要懂 Linux 的工程师

"在做双 Bank 升级"

  • 字面意思：把 Flash 分成两个区，升级时写 A 区用 B 区跑，写完后切换。升级失败还能退回旧版本
  • PM 要知道：这是最安全的 OTA 方案，但会吃掉你大概一半的 Flash 容量。如果 Flash 本来就很紧，这功能加不进去

"产测脚本还没写好"

  • 字面意思：还没写自动化测试的程序。每块板子出厂前需要跑一遍功能测试，烧录固件 + 验证功能
  • PM 要知道：这是量产前最后一道关，别压缩这个阶段。不良品流到用户手里，退货成本远超多测一天的工时

6.2 给 PM 的一个"反向提问清单"

下次工程师跟你说"做不了"的时候，你可以问几个问题来判断是真不行还是条件没谈拢：

  1. "是芯片硬件不支持，还是开发时间不够？" —— 前者需要换平台，后者加人加时间就行
  2. "换一个更贵的芯片能解决吗？大概加多少成本？" —— 很多时候问题出在成本约束而不是技术
  3. "这个功能在其他同行产品上是做出来的，他们是怎么做的？" —— 暗示这个功能是可行的，需要找到做法
  4. "如果功能上简化一点（比如降分辨率、延长响应时间），能做吗？" —— 给工程师一个降级的选项而不是一刀切的"做不了"
  5. "这个风险的概率有多大？最坏情况是什么？" —— 很多工程师说"有风险"指的是"万一出问题会很麻烦"，但实际概率很低

OTA 通过无线网络远程更新固件 产品上市后能不能修 bug、加功能，就看有没有 OTA
BOM 物料清单，所有元器件 + 结构件的列表和单价 产品定倍率和零售价的基础
PCB 印刷电路板，承载所有元器件和电路走线的板子 PCB 的层数、面积直接影响整机成本。打板需要 1-2 周
原理图 电路连接的设计图，电气工程师的产出物 评审原理图时 PM 需要确认的是：接口位置、外部连接器的类型
Layout 把原理图变成 PCB 上实际走线布局 Layout 完成后就不能改电路了，改一次 = 重新打板 1-2 周
EVT/DVT/PVT 工程验证 → 设计验证 → 产线验证，产品成熟度递增的三个阶段 这是硬件产品研发的里程碑，每个阶段结束前不要推进下一个阶段

最后的话： 这份指南不是一次读完的。下次你在项目会上听到"Flash 打满了"、"低功耗还没调"、"要换大封装"——翻到对应的章节，找到工程师说这句话时真正在焦虑的东西，然后你知道自己能做什么。这就是它最大的用处。

硬件 PM 和软件 PM 最大的区别

第二章：需求来了怎么办

2.1 需求接收 —— 不只是记下来

当老板/客户/市场丢过来一个需求时（比如"我们要做一个智能宠物喂食器"），不要直接跟开发说。首先做一个技术预评估：

第一步：拆解功能清单

把模糊的需求拆成具体的功能项。比如"智能宠物喂食器"拆成：

□ 定时出粮 —— MCU + 步进电机 + RTC
□ 手机远程控制 —— WiFi/BLE 模块 + 云平台
□ 余粮检测 —— 超声波/红外传感器
□ 出粮记录查看 —— 云存储 + App 端历史数据
□ 低电量提醒 —— ADC 检测电池电压
□ 防卡粮机制 —— 电机堵转检测

第二步：对每一功能项做四维评估

维度 你要问的问题
技术可行性 这个功能当前方案能不能做？需要新芯片吗？
成本影响 加这个功能 BOM 涨多少？
时间影响 开发这个功能需要多久？会不会影响其他功能？
风险等级 方案成熟吗？有现成的方案还是需要研发？

第三步：输出"技术可行性摘要"

给开发团队一份简短的需求评估文档，格式如下：

## 需求：智能宠物喂食器 V1

### 关键功能及评估

| 功能 | 方案 | 可行性 | 成本影响 | 估算工时 | 备注 |
|------|------|--------|---------|---------|------|
| 定时出粮 | 步进电机 + RTC | ✅ 成熟方案 | +15 元 | 2w | |
| 手机控制 | ESP32 WiFi + MQTT | ✅ 有现成方案 | +20 元 | 4w | 云平台费用另计 |
| 余粮检测 | 超声波模块 HC-SR04 | ⚠️ 建议改红外 | +3 元 | 1w | 超声波在粮仓内反射不准 |
| 低电量提醒 | ADC 分压检测 | ✅ 零成本 | — | 0.5w | |
| 防卡粮 | 电机电流检测 | ⚠️ 需要额外电路 | +5 元 | 2w | 结构设计配合也很关键 |

### 总体评估
- 预估 BOM 成本：60-80 元
- 预估开发周期：12-16 周
- 主要风险点：WiFi 配网体验、防卡粮机械结构

2.2 和开发团队开会时，带什么去

硬件评审会和软件站会完全不同。硬件评审需要明确交付：

评审类型 参与人 PM 要带什么
立项评审 全员 功能清单 + 竞品分析 + 成本目标 + 时间线
方案评审 硬件 + 嵌入式 功能清单 + 接口需求表 + 功耗要求
原理图评审 硬件 外部接口定义 + 电源需求 + 尺寸约束
结构评审 硬件 + 结构 外观设计图 + 按键/屏幕/接口位置图
试产评审 全员 完整 BOM + 测试用例 + 认证要求

每个评审会 PM 要问的三个核心问题：

  1. 当前方案有没有技术盲区？（没见过的新方案需要 POC 验证）
  2. 现有的时间线里，风险最大的是哪一段？（打板？联调？认证？）
  3. 如果今天要砍一个功能，砍哪个影响最小？（永远准备好 Plan B）

2.3 需求变更处理流程

硬件产品的需求变更必须走正式流程，不能口头一句"加个功能"：

收到变更请求
    ↓
PM 评估变更影响（技术/成本/时间/风险）
    ↓
更新需求文档 / PRD
    ↓
和开发团队确认可行性
    ↓
更新排期和 BOM
    ↓
同步给相关方（采购/生产/市场）

变更常见类型和应对：

变更类型 典型示例 影响 PM 对策
加传感器 "能不能加个温度检测" 可能只需加 I2C 传感器，成本低 如果引脚有余量，通常可行
改通信方式 "蓝牙改 WiFi" 可能换芯片平台，全部重做 成本很高，尽量在方案阶段定死
加显示屏幕 "加个 OLED 屏" 增加引脚占用、Flash、功耗 评估引脚和 Flash 余量
改外壳颜色 "要磨砂黑" 可能只是换料 如果模具已开，换颜色可能涉及新模具
改电池容量 "续航要翻倍" 电池体积变大，外壳可能需要改 结构上预留空间很重要

第三章：怎么和硬件团队高效沟通

3.1 团队里各角色在做什么

角色 核心工作 产出的东西 PM 需要从他们那里知道什么
硬件工程师 芯片选型、画原理图、PCB Layout 原理图、BOM、PCB 文件 芯片交期、引脚余量、BOM 成本
嵌入式工程师 写固件、调外设驱动 固件代码、烧录文件 Flash/RAM 占用率、开发进度
结构工程师 设计外壳、散热、装配 3D 模型 CAD 文件 尺寸约束、模具费用、开模周期

测试工程师 功能测试、可靠性测试 测试报告、问题清单 测试覆盖率、复现率最高的 bug

3.2 硬件 PRD 长什么样

软件 PRD 和硬件 PRD 的最大区别：硬件 PRD 必须包含接口定义和物理约束。

一个硬件 PRD 的骨架：

# PRD：智能宠物喂食器 V1

## 1. 产品概述
- 产品定位：中端智能宠物喂食器，目标售价 299 元
- 目标用户：上班族养猫/狗人群
- 竞品参考：小米/小佩同类产品

## 2. 功能清单（按优先级）
P0（必须有）：
- 定时出粮（每天最多 4 个时段）
- 手机远程控制出粮（WiFi）
- 余粮低告警
- 防卡粮

P1（应该有）：
- 出粮记录查看（7 天历史）
- 低电量提醒

P2（最好有）：
- 语音录制播放
- 摄像头查看宠物进食

## 3. 硬件接口需求
| 接口 | 连接什么 | 数量 | 电压/协议 |
|------|---------|------|----------|
| GPIO | 步进电机驱动板 | 4 | 3.3V |
| PWM | 舵机（备用方案） | 1 | 50Hz |
| UART | WiFi 模块 | 1 | 115200bps |
| I2C | 红外传感器 | 1 | 100kHz |
| ADC | 电池电压检测 | 1 | 0-4.2V |
| GPIO | 指示灯 LED | 2 | 3.3V |

## 4. 功耗要求
- 待机功耗：< 50μA（电池供电时）
- 工作功耗：< 500mA（电机转动时）
- 电池容量：18650 一节（2600mAh）
- 目标续航：30 天（每天出粮 2 次）

## 5. 物理尺寸
- 整机尺寸：200×150×250mm
- 粮仓容量：2L
- 按键/指示灯位置：顶部面板
- 屏幕：无（V1 版本不带屏幕）

## 6. 通信协议（App ↔ 设备）
- 配网方式：蓝牙辅助配网
- 云端协议：MQTT
- 本地控制协议：UART AT 指令集

给 PM 的建议： 接口需求表是硬件 PRD 里最重要的一张表。它直接决定了：

  • 芯片需要多少个引脚（决定了芯片价格和封装）
  • PCB 上需要布多少线（决定了 PCB 尺寸和层数）
  • 外壳上需要开多少个孔（决定了模具复杂度）

3.3 正确问问题的方式

PM 问问题的措辞决定了工程师会怎么回答。对比两组对话：

❌ 低效版：

PM："这个功能能不能做？"
工程师："做不了。"
PM："为什么做不了？"
工程师："这个芯片不支持。"
PM："那换一个芯片呢？"
工程师："要评估一下。"
—— 对话死了

✅ 高效版：

PM："这个功能需要 MCU 支持 I2S 音频接口，我们现在用的芯片有吗？"
工程师："STM32F103 没有 I2S，要换到 F4 系列才有。"
PM："F4 系列的封装和 F103 兼容吗？成本差多少？"
   —— 对话变成了具体的技术决策

几个 PM 可以用的典型追问：

  1. "这个是芯片硬件不支持，还是开发时间不够？" —— 前者要换芯片，后者加人加时间
  2. "如果功能上降级（比如降低采样率/减少存储天数/延长响应时间），能做吗？" —— 给出降级选项
  3. "这个风险的触发概率有多大？最坏情况是什么？" —— 区分"有可能出问题"和"大概率会出问题"
  4. "如果成本预算加 10 元，能解决吗？" —— 很多时候问题出在成本约束而不是技术可行性

第四章：用 Figma 做交互原型

4.1 Figma 基础 —— 15 分钟上手

Figma 是一款基于浏览器的 UI 设计工具。不需要安装（但有桌面版），免费版足够个人使用。

核心概念（用开发的语言说）：

Figma 术语 对应开发概念
Frame 相当于一个 div/view/screen，定义画布尺寸
Layer 相当于 HTML 元素，一个 Frame 里可以套无数层
Component 相当于可复用的 React 组件/Vue 组件
Auto Layout 相当于 flexbox，自动排列子元素
Constraints 相当于相对定位/响应式布局
Prototype 交互连线，相当于路由跳转
Variants 同一个组件的不同状态（hover/pressed/disabled）

第一步：创建一个 Frame

  1. 打开 Figma，点 + 号 → 选手机模板（iPhone 14 / 15）
  2. 这就是你的第一个屏幕画布

第二步：拖拽基础元素

  • 用快捷键 R 画矩形（按钮）
  • 用快捷键 T 打字（标签）
  • 用快捷键 O 画圆形（图标背景）
  • 从右侧面板调整颜色、圆角、阴影

第三步：用 Auto Layout 选中一个 Frame，按 Shift+A 开启 Auto Layout。它的作用：

  • 子元素自动水平或垂直排列
  • 间距统一控制
  • 内容增减时自动撑开

这是 Figma 里最重要、最值得花 30 分钟练习的一个功能。

4.2 硬件产品的交互设计特点

硬件产品的 App 交互和纯 App 不一样：

维度 纯 App 设计 硬件配套 App 设计
页面数量 可能几十上百个页面 通常 5-15 个（配置/控制/设置/关于）
核心交互 滑动、点击、长按 开关、滑块、定时、状态展示
设备连接 无 配网/绑定/OTA 升级是核心流程
离线状态 不常见 必须考虑设备离线、断连、低电量
错误处理 弹窗提示 要考虑设备无响应/指令超时/固件更新失败

硬件 App 的典型页面结构：

引导页（首次使用）
    ↓
设备配网（WiFi 配网 / 蓝牙绑定）
    ↓
主控制页（开关/调节/模式切换）
    ↓
设置页（定时/通知/固件升级/关于）
    ↓
设备列表（如果有多个设备）

4.3 从线框图到高保真原型

第 1 步：手绘线框图（纸上）

不要一开始就打开 Figma。先用笔和纸画出每个页面的布局。关注：

  • 信息的优先级（什么功能放在最显眼的位置）
  • 操作路径（用户点完第一步之后去哪里）
  • 物理设备的对应关系（屏幕上的开关和真实设备的按键对应吗）

第 2 步：Figma 低保证线框图（灰度）

用灰度方块代替所有 UI 元素。不配色、不选字体、不用真实图片。目的是验证布局和交互流程。

第 3 步：Figma 高保真原型

加上真实配色、字体、图标、图片。用 Prototype 模式连接页面之间的跳转。

给硬件 PM 的省钱建议： 如果你的团队没有专职设计师，这套流程里做到第 2 步（灰度线框图）就够了。灰度线框图足够给开发讲清楚"这个页面长什么样、按了这个按钮去哪"。真正的高保真可以等产品验证通过后再交给设计师精修。

4.4 Figma 的组件化思维

做硬件 App 时，以下组件值得你先做好然后复用：

通用组件库（建议第一周就建好）：
├── 设备卡片（设备名、状态、信号强度）
├── 开关控件（开/关/正在切换中）
├── 滑块（亮度/音量/温度调节）
├── 定时卡片（时段起始/重复/开关）
├── 状态标签（在线/离线/升级中/故障）
├── 配网流程页（搜索设备 → 连接 → 配网成功/失败）
├── 底部导航栏
└── 加载/空状态/错误状态

在 Figma 里把这些做成 Components（Component 创建后，你改一个实例，所有实例同步更新），后续拼接页面就像搭积木。

第六章：实战案例 —— 从零设计一个智能硬件产品

场景

你是 PM，公司决定做一个"智能胶囊咖啡机"（对标 Nespresso 但可以手机控制）。从需求到开发交付，你该怎么走？

第 1 周：需求定义

输出： 一页纸的功能清单 + 竞品分析

P0（必须有）：
- 手机控制萃取（浓度/温度/杯量）
- 胶囊识别（NFC 读取胶囊信息）
- 水箱水位检测
- 废水盘满提醒

P1（应该有）：
- 定时预约
- 个人口味配置文件（3 个预设）
- 缺水/缺胶囊自动告警

P2（最好有）：
- 咖啡豆余量统计
- 冲煮历史记录

第 2 周：技术预评估 + 选型

输出： 技术可行性评估表

你和硬件工程师开会对了一遍：

功能 方案 引脚 评估
NFC 胶囊识别 I2C NFC 模块 +2 引脚 ✅ 成熟方案
水泵控制 MOSFET + PWM +2 引脚 ✅
水位检测 电极式 + ADC +1 引脚 ⚠️ 干烧保护要单独做
加热控制 SSR + 温度传感器 +2 引脚 ✅ NTC 温度检测
手机控制 ESP32（自带 WiFi/BT） UART ✅ ESP32 自带双模

结论：一片 ESP32 能覆盖所有功能，不需要 MCU + 独立 WiFi 模组。

第 3 周：交互原型

输出： Figma 灰度线框图（5 个核心页面）

1. 配网引导页 —— 首次使用流程
2. 主控制页 —— 选择口味/杯量/浓度
3. 预约定时页 —— 设置冲煮时间
4. 设备状态页 —— 水箱/胶囊/废水盘状态
5. 个人设置页 —— 口味预设

用 AI 生成初始布局参考 → 在 Figma 调整成可交付的线框图。

第 4 周：PRD + 评审

输出： 完整 PRD

包含：

  • 功能清单（P0/P1/P2）
  • 接口需求表（哪个外设接哪个引脚）
  • 功耗要求（目标：待机 < 1W）
  • 物理尺寸约束
  • App 交互流程图
  • 认证要求（3C / FCC / ETL）

第 5 周起：进入开发

你的角色从"定义"切换到"跟进"：

  • 每周和硬件工程师检查原理图进度
  • 每两周看一次固件 Demo
  • 跟踪芯片采购交期
  • 提前联系认证机构
  • 准备产测方案

原理图评审

  • 所有外设的引脚没有冲突

固件评审

  • 认证报告拿到了吗
